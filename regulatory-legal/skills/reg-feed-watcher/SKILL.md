# /reg-feed-watcher

1. 读 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` → 关注清单、重要性阈值、信息源配置。
2. 使用以下工作流。
3. 拉取每个信息源。按重要性过滤。
4. 输出：新内容，按重要性档分类。

---

## 目的

拉取信息源，按重要性过滤，输出留下的。过滤本身就是价值——未过滤的信息源只是噪音。

## 加载上下文

`~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` → 关注清单、重要性阈值、信息源配置、摘要输出路径（如已设）。

`references/source-catalog.md`（在此技能目录中）→ 各类信息源的整理目录（覆盖中央层级、地方层级、行业协会、二级来源等）。在配置新源或用户的关注清单存在覆盖缺口时使用（参见步骤 0）。

## 工作流

### 步骤 0：覆盖检查（拉取之前）

在运行拉取之前，对照 `references/source-catalog.md` 比较关注清单 + 信息源配置：

- 用户按关注清单关注哪些类别（中央监管/地方监管/行业协会/司法）？
- 哪些类别配置的源极少或没有？

如果存在明显缺口——例如，用户在关注清单监视"网信办"，但信息源配置里只有一个 `cac.gov.cn`，遗漏地方网信办、相关协会、最高院数据条线——在摘要顶部一次性提出：

> **信息源覆盖缺口提醒：** 你的关注清单包含 [类别]，但仅配置了 [N] 个源。源目录在此类别中列出了 [X] 个选项（例如，[前 2-3 个名称]）。要我建议添加吗？运行 `/regulatory-legal:cold-start-interview --redo` 进行更新，或直接编辑 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md`。

不要反复唠叨相同的缺口——如果用户明确表示"暂时跳过地方监管"，请尊重。在 CLAUDE.md 中记录状态以便保持。

### 步骤 1：拉取

从所有配置的信息源档拉取。**优先级从高到低**：

**第 1 级——元典开放平台 MCP（首选）**

如果元典 MCP 已配置且可用：通过 `mcp__yuandian-law__*` 工具按上次核查日期查询新法规 + 案例 + 行政处罚，过滤关注清单监管机构。

**为什么元典是首选**：
- 已结构化数据（标题、文号、发布机关、生效日期等字段化）
- 覆盖中央层级法律/行政法规/部门规章/司法解释/指导性案例/行政处罚
- 引用核验有据——元典对应一手出处
- 站点改版不影响（API 稳定）

若元典 MCP 不可用（未配置 API key、调用失败）：在简报顶部明确说"元典 MCP 不可用，本次降级到栏目页 + 网页检索"，**不要静默降级**。

**第 2 级——栏目页轮询（主力方案，覆盖元典未覆盖的源）**

对 CLAUDE.md 中 `## 信息源配置` 标 `栏目页` 类型的源：

- **URL 时效检查**：读配置中的 `验证日期` 和 `状态` 字段。若状态为 `❌ 待补`，跳过轮询，在简报底部注："[监管] 栏目页状态为待补，未执行轮询。跑 `cold-start-interview --redo` 重新搜索验证有效 URL。"若验证日期为空或超过 **90 天**，先尝试浏览器访问验证 URL 仍可达；如不可达，执行一次联网搜索重新确认当前有效 URL，更新配置后再轮询。
- **获取栏目页**：抓取政策列表页（`<title, link, publish_date>` 结构）
- **指纹对比**：与上次见过的 URL 集合对比，识别新增
- **仅将新增项**进入步骤 2 分类

常用中国监管机构公开来源参考：
| 监管机构 | 覆盖领域 |
|---|---|
| 国家法律法规数据库 flk.npc.gov.cn | 全国性法律、行政法规 |
| 中国政府网 gov.cn | 国务院政策文件 |
| 中国人民银行 pbc.gov.cn | 支付、金融市场、跨境 |
| 国家金融监督管理总局 nfra.gov.cn | 银行、保险、消金、信托 |
| 中国证监会 csrc.gov.cn | 证券、基金、上市公司 |
| 国家网信办 cac.gov.cn | 数据、个人信息、算法、生成式 AI |
| 国家市场监管总局 samr.gov.cn | 反垄断、广告、消费者权益 |
| 工业和信息化部 miit.gov.cn | 通信、电子、APP 治理 |

**⚠️ 注意**：上表**不含**可直接使用的栏目页 URL。所有具体栏目页 URL 必须在 cold-start 时通过 `WebSearch` + 浏览器验证确认当前有效性后才能登记到 CLAUDE.md。政务网站改版频繁，静态列表易失效。

**第 3 级——付费订阅源（如果配置）**

- **元典开放平台 MCP（付费订阅）：** 查询自上次核查日期以来的更新，过滤到关注清单监管机构。
- **北大法宝/威科先行 MCP（如已配置）：** 同上。

跨档去重——同一文档可能出现在多个源中。优先为丰富输出使用最丰富的源。

**不得静默补充。** 如果信息源拉取为关注清单中的监管机构返回很少或没有结果，报告找到的内容并停止。不要在未询问的情况下从网络搜索或模型知识填补缺口。说："信息源核查从 [命中的监管机构] 返回了 [N] 项。对于 [监管机构/主题]，覆盖似乎稀薄。选项：(1) 扩大日期窗口，(2) 尝试不同的源或 MCP，(3) 搜索网络——结果将标记 `[web search — verify]`，在依赖之前应根据发行机构官网进行检查，或 (4) 停在这里。你想要哪个？"律师决定是否接受较低置信度的来源；Claude 不为他们决定。

**来源归属。** 为每个引用和监管项目标记其来源：`[国家法律法规数据库]`、`[<监管机构> 栏目页]`、`[元典]` 或通过连接器检索的项目的特定 MCP 工具名称；`[web search — verify]` 用于来自网络搜索的项目；`[model knowledge — verify]` 用于从模型训练数据中浮现的项目；`[user provided]` 用于手动粘贴的项目。带有 `verify` 标签的项目比工具检索的项目具有更高的伪造风险，应首先检查。永远不要剥离或折叠标签——它们是用户了解要核验哪些引用的最快信号。

**二级来源。** 律所客户提醒、行业公众号（金融监管研究院、互金通讯社等）、法律评论（互金通讯社等评论员/聚合商）报告主要监管行动，但不是主要来源。将从这些信息源提取的任何项目标记为 `[二级来源]` 以及来源标签——例如，`[金融监管研究院公众号] [二级来源]`。在简报中，当二级来源项目描述监管机构行动时，添加注释："→ 追溯到一手出处：[如果已知，链接到监管机构官网，否则'在依赖之前在 <监管机构>.gov.cn 上查找']。不要仅凭二级来源项目的强度将其归类为"始终重要"——在找到一手出处之前将其降低一个档。

**第 4 级——手工录入**

如果用户粘贴了监管文本或摘要而不是从计划的信息源核查调用：将粘贴的内容视为单个项目，跳至步骤 2 进行分类，并记录来源为"手工录入"。不需要信息源拉取。无论订阅状态如何，此路径都有效。

拉取后记录核查时间戳。下一次计划运行从此处向前拉取。

### 步骤 2：分类

每个项目根据 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 获得重要性档：

| 项目类型 | 与阈值匹配 |
|---|---|
| 最终部门规章/行政法规 | 通常"始终重要" |
| 征求意见稿 | 通常"值得复核"——并始终记录评议截止日期 |
| 信息征询（类似 RFI） | 与征求意见稿相同——规则前，无合规义务，但评议截止日期是真实的。仅作为预定位分析路由到 `/regulatory-legal:policy-diff` |
| 行政处罚通报（同业） | 重要；相关实践匹配→值得复核；两者都不→仅供参考 |
| 指引性文件（非约束） | 值得复核 |
| 新司法解释 | 通常"始终重要"（有法律约束力，作裁判依据） |
| 指导性案例 | 值得复核（作裁判理由，不作裁判依据，效力位阶不同） |
| 答记者问/监管发声/讲话 | 仅供参考或根据阈值跳过；强制加 `[非正式监管口径——需核验]` |
| 行业自律新规 | 值得复核 |
| 国家/团体标准 | 值得复核 |

**征求意见稿处理——具体说明。** 规则前项目在一个重要方面与最终规则不同：它们不改变法律，但它们确实带有评议截止日期并且它们发出监管机构的方向信号。将它们视为一个单独的分支：

- **不要**将征求意见稿归类为"始终重要"——在规则发布之前合规影响为零。
- **如果征求意见稿的任何问题领域触及关注清单的始终重要类别（例如，金融服务关注清单中关于开放银行的征求意见稿），则**归类为值得复核。
- **将评议截止日期记录到** `~/.claude/plugins/config/claude-for-legal/regulatory-legal/comment-tracker.yaml`，并带有 `item_type: 征求意见稿`，以便下游台账可以将其与合规差异区分开来。
- **在简报条目中包含一行**，明确指出："规则前。评议截止日期 [日期]。仅作为预定位分析路由到 `/regulatory-legal:policy-diff`（尚无合规差距）。"
- **路由到评议台账，而不是内规差异台账。**

**评议截止日期处理：**

对于以任何高于"跳过"档分类的每个征求意见稿：
- 提取评议截止日期
- 如果在 CLAUDE.md 中启用了评议跟踪：附加到 `comment-tracker.yaml`，状态为"未决定"，以及来自 CLAUDE.md 的默认评议决策所有者
- 在输出条目中包含评议截止日期

### 步骤 3：丰富

对于每个高于仅供参考档的项目：

- 一行摘要（改变了什么）
- 为什么这里可能重要（相关性钩子——"这是关于 [你所做的实践]"）
- 指向源的链接
- 生效日期或评议截止日期（如适用）

不要单独总结仅供参考项目——只计算它们。

## 输出

摘要默认进入聊天。**同时将其写入可共享文件**，每当输出包含一个或多个高于仅供参考的项目时，除非用户的 CLAUDE.md 明确设置 `摘要输出 → 仅聊天`。

**文件输出行为：**

1. 在 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 中查找 `摘要输出路径`。如果已设置，则写入此处。如果未设置的默认值：`~/regulatory-legal-digests/reg-digest-YYYY-MM-DD.md`。
2. 如需要则创建父目录。
3. 将完整摘要写入为 Markdown（与聊天输出内容相同，包括工作产品标题、源标签和核验引用页脚）。
4. 如果该路径今天已存在文件，则附加带有时间戳子标题的新部分而不是覆盖——同一天可能会看到多次运行。
5. 写入后，告诉用户："摘要已写入 `<path>`。可按原样共享，或使用 Pandoc 转换为 .docx：`pandoc <path> -o <path>.docx`。"
6. 如果写入失败，回退到仅聊天输出并说明——不要静默删除文件请求。

## 配置依赖的后备方案

此技能从 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 读取关注清单、重要性阈值和信息源配置。当所需值仍然是 `[PLACEHOLDER]` 或为空时，在输出中具体说明：

- **关注清单为空：** 停止并说"你配置中的关注清单为空。我不知道要监视哪些监管机构，无法拉取信息源。运行 `/regulatory-legal:cold-start-interview --redo` 或编辑 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 并添加至少一个监管机构。"
- **重要性阈值为空：** 回退到默认档并附加："此输出使用了默认重要性档，因为你的配置没有设置自定义阈值。使用 `/regulatory-legal:cold-start-interview --redo` 或通过编辑 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 进行调整。"
- **信息源配置为空：** 仅运行公开网页源（元典 MCP 如可用，否则手工录入）并附加："此输出仅使用了公开网页源，因为你的配置没有列出任何经验证的栏目页 URL 或付费订阅源。使用 `/regulatory-legal:cold-start-interview --redo` 添加信息源。"

当相关值已填充时，不说明配置。

如果没有任何高于仅供参考的项目："一切正常。[N] 个仅供参考项目，无需注意。"

## 移交

- **到 policy-diff：** 任何具有可能政策影响的"始终重要"项目→提议运行比对。
- **到 gap-surfacer：** 如果比对发现差异→已跟踪。
- **到 comment-tracker：** 任何分类为高于"跳过"的征求意见稿→如果启用跟踪，评议截止日期自动记录。

## 用下一步决策树结束

根据 CLAUDE.md `## 输出` 以下一步决策树结束。根据此技能刚刚生成的内容自定义选项——五个默认分支（起草 X、升级、获取更多事实、观察等待、其他事情）是起点，而不是锁定。树就是输出；律师选择。

## 此技能不做什么

- 完整阅读每个项目。它进行分类和丰富；深度阅读适用于通过过滤的项目。
- 更改重要性阈值。如果过滤器错误，编辑 CLAUDE.md。
- 需要元典或其他付费工具。公开网页源是基线；付费工具增加深度。
