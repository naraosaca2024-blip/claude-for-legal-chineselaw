# Gap Surfacer（内规差异呈现）

> 所有者通知：默认开启。要让所有者退出，将 `owner_dm` 留空。

## 每次发送确认——无例外

在发送任何 DM 消息（分配通知、过期提醒、批量通知、状态报告）之前：

1. 向用户准确显示你即将发送的内容以及发送给谁："我即将将此发送给 [N] 个人：[预览]。"
2. 等待明确的 yes。
3. 如果消息包含任何引用、截止日期或合规结论，添加："⚠️ 此消息中的引用未经验证——我在发送之前不确认它们是当前的。你想让我添加'在行动之前核验'行吗？"
4. 没有确认绝不发送。不是按计划。不是批量。不是因为昨天发送过。

在没有确认的情况下自动发送是此插件中最不可逆的操作，发送此插件自己的页脚声称可能错误的内容，给无法检查的人。这种组合不能跳过审查。

## 事项上下文

**事项上下文。** 检查执业级 CLAUDE.md 中的 `## 事项工作区`。如果 `已启用` 是 `✗`（内部法务用户默认），跳过此段的其余部分——技能使用执业级上下文，事项机制不可见。如果启用且没有活动事项，询问："这是哪个事项？运行 `/regulatory-legal:matter-workspace switch <slug>` 或说 `practice-level`。"加载活动事项的 `matter.md` 以获取事项特定的上下文和覆盖。将输出写入事项文件夹 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/matters/<matter-slug>/`。除非 `跨事项上下文` 是 `on`，否则绝不读取另一个事项的文件。

## 目的

内规差异被发现然后被遗忘。此技能跟踪它们直到关闭，并通知负责关闭它们的人。

## 跟踪器

位于 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/gap-tracker.yaml`：

> **关于 comment-tracker.yaml 的说明：** `~/.claude/plugins/config/claude-for-legal/regulatory-legal/comment-tracker.yaml` 是由 comments 技能拥有的兄弟文件。它由 reg-feed-watcher（自动记录征求意见稿）和 comments 技能（跟踪用户发起的评议决定）写入。此技能不读取或交叉引用它。如果你修改 comment-tracker schema，请更新两个实际消费者。

```yaml
gaps:
  - id: GAP-001
    requirement: "[法规要求的内容]"
    regulation: "[名称 + 引用]"
    policy_affected: "[名称或'需要新政策']"
    gap_type: "partial"  # none | partial | full | new-policy | watch | comment-decision
    owner: "[来自政策索引的名称]"
    owner_dm: "[飞书/钉钉/企业微信用户 ID 或句柄，如果已知]"
    opened: 2026-03-01
    due: 2026-06-01  # 法规生效日期、内部截止日期或评议截止日期
    status_verified: true  # 如果上游 policy-diff 无法确认规则生效则为 false；未验证的项目永远不会达到 🔴 过期
    status: "open"  # open | in-progress | closed | risk-accepted
    notified: false  # 在发送分配通知后设置为 true
    resolution: ""  # 关闭时填充
```

**永远不要将内规差异归类为未验证规则的过期。** 🔴 过期分类意味着"我们错过了有约束力的截止日期。"如果规则状态未验证（policy-diff 设置 `status_verified: false`，或规则 >12 个月/超过其适用日期且无货币确认），截止日期可能没有约束力。使用 🟡 "需要审查"并注意："如果此规则按发布生效，这将过期 [N] 天。在升级之前验证规则状态。"将未验证规则项目路由到 `watch`，而不是活动的过期/即将到期存储桶；`watch` 重新评估节奏在项目可以重新作为合规内规差异浮出水面之前强制进行规则状态检查。

**`gap_type` 语义：**

| 值 | 含义 | 典型提醒节奏 |
|---|---|---|
| `none` | 政策已覆盖要求。仅记录用于审计跟踪。应该很少——如果大多数条目是 `none`，差异可能针对错误的政策运行。 | 无自动提醒。 |
| `partial` | 政策涉及主题但不完全覆盖新要求。需要修正。 | 截止前 30 天。 |
| `full` | 政策与新要求相矛盾或默默省略。需要重写或新章节。 | 截止前 30 天。 |
| `new-policy` | 没有现有政策涵盖此内容。需要起草政策。 | 截止前 30 天。 |
| `watch` | 前瞻性项目——征求意见稿、信息征询、尚未最终的拟议规则。今天没有合规义务；政策工作等待最终规则。`due:` 是重新访问日期（通常是正式规则预期日期或一年期限），而不是合规截止日期。 | 无自动提醒；在正式规则发布时或重新访问日期重新评估。 |
| `comment-decision` | 规则前评议决策待定——团队正在决定是否就征求意见稿提交意见。`due:` 是评议截止日期。 | 截止前 21 天（比合规差异更紧，因为意见起草窗口更短）。 |

`watch` 或 `comment-decision` 条目不是合规差距——它是 watch 技能和 comments 技能产生的规则前项目的跟踪工件。在状态报告中在自己的存储桶中呈现它们，以便在上午阅读的法务可以一目了然地区分哪些项目是"在监管机构注意到之前修复此内容"与"关注此内容"。

## 模式

### 模式 1：从 policy-diff 摄取

当 policy-diff 发现内规差异时，将它们附加到 gap-tracker.yaml。去重——相同要求 + 相同政策 = 相同差异，不要重复计算。

**摄取后，通知所有者：**

如果飞书/钉钉/企业微信 MCP 可用且设置了 `owner_dm`：

向差异所有者发送 DM——但仅在此文件顶部的每次发送确认之后。向用户预览消息，等待明确的 yes，然后发送：

```
📋 分配给你的新内规差异

差异：[GAP-ID] — [要求，一句话]
法规：[名称 + 链接]
受影响的政策：[政策名称或"需要新政策"]
截止：[法规生效日期]

查看完整差异台账：/regulatory-legal:gaps
```

发送后在跟踪器条目中设置 `notified: true`。

如果飞书/钉钉/企业微信 MCP 不可用：在状态报告中注明未发送所有者通知并标记手动跟进。

```markdown
**在依赖之前验证引用。** 此跟踪器中的法规引用是上游 AI 生成的（由 reg-feed-watcher 和 policy-diff），尚未与主要来源核对。在关闭或风险接受差异之前——或在证明、董事会报告或监管机构回应中引用差异——根据元典、北大法宝、威科先行或发行机构的官网确认基础规则。AI 生成的法规引用有时会被伪造、误引用或过时。从上游携带的源标签（例如，`[国家法律法规数据库]`、`[web search — verify]`）显示每个引用的来源；带有 `verify` 的标签具有更高的伪造风险，应首先检查。在呈现差异时绝不剥离标签。
```

## 配置依赖的后备方案

此技能从 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 读取差异响应所有者和升级路径。当它需要的值为空或仍然是 `[PLACEHOLDER]` 时：

- **缺少差异响应分流者：** 将分配保留开放并附加到输出："`## 内规差异响应流程` 中未设置分流者。使用 `/regulatory-legal:cold-start-interview --redo` 或通过编辑 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 分配一个，以便新差异得到路由。"
- **新摄取差异的所有者未知（政策库中没有所有者）：** 使用 `owner: [unassigned]` 记录差异并附加："[N] 个差异在没有所有者的情况下被摄取，因为政策库没有为受影响的政策命名一个。填写政策库中的所有者列以路由它们。"
- **过期重大差异缺少升级路径：** 仍将其报告为过期，并附加："未为重大过期差异设置升级路径。使用 `/regulatory-legal:cold-start-interview --redo` 或通过编辑 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 配置它。"

当值已填充时，不说明配置。

**截止日期提醒逻辑（在状态报告和计划代理期间运行）：**

提醒节奏是 `gap_type` 的函数——合规差异获得 30 天提前通知，comment-decision 项目获得 21 天（更紧，因为起草窗口更短），watch 项目没有自动提醒（在正式规则发布时重新评估）。

对于每个状态为"开放"或"进行中"的差异：
- `partial`、`full`、`new-policy`、`none`：如果截止日期在 30 天内且在过去 7 天内未发送提醒，预览 DM（主题"⏰ 提醒：内规差异将在 [N] 天内到期"）并在发送前等待每次发送确认。
- `comment-decision`：如果评议截止日期在 21 天内且在过去 7 天内未发送提醒，预览 DM（主题"💬 评议决策截止日期在 [N] 天内"）并在发送前等待每次发送确认。
- `watch`：无自动提醒。在审查台账或为同一法规记录征求意见稿时重新访问。
- 如果合规差异的截止日期已过：在报告中标记为过期并预览 DM——在发送前等待每次发送确认。
- 如果 `comment-decision` 项目的评议截止日期已过且未提交意见：标记为过期，预览 DM（等待每次发送确认），并要求所有者更新为 `risk-accepted`（有意不评论）或 `closed`（已提交意见）并附带说明。
- 在跟踪器中记录提醒时间戳以避免重复唠叨。
- 批量提醒仍需要每次发送确认——预览"你即将发送 12 条 DM"并等待 yes 计数；静默触发批量不算。

### 后果行动关卡（证明合规）

**在将差异关闭为已解决或产生任何证明符合监管要求的输出（内部证明、董事会报告、审计回应、监管机构回应）之前：** 阅读 `~/.claude/plugins/config/claude-for-legal/regulatory-legal/CLAUDE.md` 中的 `## 谁在使用这个`。如果角色是 **非律师**：

> 证明合规——或将差异关闭为已解决——具有法律后果。如果后来证明是错误的，证明可以被用来对抗公司，过早的关闭会使风险未得到解决。你是否与律师审查过此内容？如果是，继续。如果否，以下是带给他们的简报：
>
> - 差异（要求、来源、政策差异发现的内容）
> - 提议的解决方案涵盖和不涵盖的内容
> - 任何残留差异或歧义
> - 未解决的问题和待议事项
> - 可能出什么问题（过于宽泛的证明、未解决的残留义务、不一致的先前立场）
> - 向律师问什么（这是否真正关闭；我们应该以理由风险接受而不是；我们需要外部律师同意吗）
>
> 如果你需要找到律师：律师协会的引荐服务是最快的起点（上海市律师协会、北京市律师协会等地方律协；中华全国律师协会有指引）。

在没有明确的 yes 的情况下，不要将差异标记为关闭或在此关卡之后产生合规证明。状态报告和跟踪视图不需要关卡。

### 模式 3：关闭差异

```
/regulatory-legal:gaps --close GAP-001
解决方案："政策已更新 v2.3，批准于 [日期]"
```

将状态更新为关闭，记录解决方案和关闭日期。

### 模式 4：风险接受差异

有时答案是"我们不打算修复这个。"这是一个有效的决定——但应该记录。

```
/regulatory-legal:gaps --accept GAP-002
理由："要求仅适用于[我们不满足的条件]。如果[触发器]则重新访问。"
接受者：[有权威的姓名]
```

状态 → risk-accepted。停留在跟踪器中（未删除）但从开放差异报告中脱落。

## 集成：reg-change-monitor 代理

代理的摘要包括差异计数和最古老的开放差异年龄。如果有任何过期，那将在摘要的顶部。代理还运行截止日期提醒检查并发送任何未完成的通知。

## 用下一步决策树结束

根据 CLAUDE.md `## 输出` 以下一步决策树结束。根据此技能刚刚生成的内容自定义选项——五个默认分支（起草 X、升级、获取更多事实、观察等待、其他事情）是起点，而不是锁定。树就是输出；律师选择。

如果台账呈现超过约 10 个开放差异，或者任何时候用户询问：提供仪表板（参见 CLAUDE.md `## 输出 → 数据密集型输出的仪表板提议`）。为此输出塑造提供——按严重性的计数、按截止日期的差异时间线，以及带有所有者、状态和最后接触日期的可排序网格。

## 此技能不做什么

- 自己关闭差异。关闭需要解决方案笔记和人类行动笔记描述的行动。
- 如果未配置通知 MCP 则发送通知。回退到在状态报告中标记。
- 每 7 天每周期的差异发送超过一个提醒。唠叨一次，而不是不断唠叨。
