# /reg-gap-analysis（数据合规差距分析）

1. 加载执业档案 → 监管范围、个人信息处理规则承诺、个人权利请求系统。
2. 运行以下工作流。
3. 范围分析：法规是否适用？（适用主体/阈值/行业）
4. 提取法规要求 → 与当前合规状态差距比较 → 差距列表。
5. 带有负责人、日期、优先级的整改计划。
6. 保存带日期的文档。即使"无差距"也要记录（以证明已做评估）。

```
/privacy-legal:reg-gap-analysis "个人信息保护合规审计管理办法"
```

```
/privacy-legal:reg-gap-analysis "网络数据安全管理条例"
```

## 差距比较

对照法规要求和当前合规状态：

| # | 合规要求 | 法规条款 | 当前状态 | 差距描述 | 风险级别 |
|---|---|---|---|---|---|
| 1 | [要求] | [条款] | [已满足/部分满足/缺失/不确定] | [差距描述] | [🔴/🟠/🟡/🟢] |
| 2 | | | | | |

**差距来源：**
- ✅ 已满足——无差距
- ⚠️ 部分满足——有改善空间
- ❌ 缺失——明确差距，须整改
- ❓ 不确定——须进一步核实当前状态

|---|---|---|---|---|---|---|
| 1 | [差距] | [具体整改措施] | [姓名/团队] | [日期] | [P0/P1/P2] | 待启动 |

**优先级定义：**
- P0（阻断）：法律明确要求且完全缺失，有重大处罚风险——须立即整改
- P1（高）：有明确法律要求，当前实践存在实质差距
- P2（中）：有法律要求，当前实践部分满足但有提升空间

## 此 skill 不做什么

- 它不确定法律是否"真的"适用——适用范围分析标记为 `[审阅]`，须律师确认。
- 它不代替合规审计——这是合规差距分析，实际合规审计须按 PIPL 54条和《个人信息保护合规审计管理办法》（2025-05-01）开展。
- 它不生成递交给监管机构的报告——这是内部分析文件。
