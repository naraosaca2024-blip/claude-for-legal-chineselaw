# /dpa-review（个人信息处理协议审查）

1. 加载执业档案 → 个人信息处理协议（委托处理协议）剧本。如果是占位符，停止并提示设置。
2. 获取协议。确定方向：我们是受托处理方（接受委托）还是个人信息处理者（委托他人）？如果不明确则询问。

# 个人信息处理协议审查

## 事项上下文

**事项上下文。** 检查执业级 CLAUDE.md 中的 `## 事项工作区`。如果 `已启用` 为 `✗`（内部用户的默认值），跳过本段的其余部分——skills 使用执业级上下文，事项机制不可见。如果已启用且没有活跃事项，询问："这是哪个事项的？运行 `/privacy-legal:matter-workspace switch <slug>` 或说 `practice-level`。"

---

## 目的

委托处理协议（个人信息处理协议）有两种类型，审查方向几乎相反。当委托方（个人信息处理者）发送他们的协议时，我们（作为受托方）在维护运营灵活性。当我们向供应商发送协议时，我们在保护我们处理的个人信息。两种审查都读取执业档案中的剧本，但从相反的行读取。

**法律依据：** 《个人信息保护法》第二十一条（委托处理的核心要求）

---

## 首先：哪个方向？

在其他一切之前，确定：

- **我们是受托处理方** → 委托方（个人信息处理者）发送他们的协议 → 读取执业档案 → "当我们是受托处理方时"剧本
- **我们是个人信息处理者（委托他人处理）** → 我们向供应商/受托方发送协议（或审查他们的）→ 读取"当我们是个人信息处理者时"剧本

如果不明确，询问。弄错这个会反转每个建议。

---

## 加载此前关于此对手方/活动的上下文

在审查之前，检查输出文件夹中关于此对手方或处理活动的先前工作：

- **相同对手方/处理活动的先前 `use-case-triage` 结果**——分类产生的风险评级和条件，此协议审查应遵守或明确偏离。
- **涵盖此对手方/处理活动的先前 PIPIA 输出**——PIPIA 可能标记了协议需要实施的风险缓解措施。
- **相同对手方的先前协议审查输出**——早期审查设定了什么可接受、什么被标记、什么被解决的期望。

如果发现先前输出，在审查中引用，说明是否与先前结论一致或偏离（并说明偏离原因）。

**将上游输出的严重性作为下限携带**——上游标为 🔴 的处理活动不能在协议审查中被静默降级为 🟢。

## 中国法律合规前置检查（在逐条审查之前）

在逐条审查之前，回答：**此处理活动是否涉及特殊监管要求？**

> **行业专项检查——首先询问：**
>
> 此处理是否涉及：
> - **金融账户信息/非公开个人金融信息**（受人民银行、国家金融监督管理总局个人金融信息保护规定约束）？如是，协议须包含：信息用途的明确限制、数据安全标准（GB/T 35273或等保要求）、数据不得用于第三方营销的限制。`[待核验最新金融监管要求]`
> - **个人健康医疗数据**（受国家卫健委等医疗数据管理规定约束）？`[待核验行业专项规定]`
> - **不满十四周岁未成年人个人信息**（PIPL 31条，须取得父母/监护人同意；需制定专门处理规则）？
> - **生物识别信息**（人脸/指纹/声纹等，属于敏感个人信息，PIPL 28条，须单独同意）？
> - **汽车数据**（受《汽车数据安全管理若干规定》约束）？`[待核验]`
> - **地图/测绘地理信息数据**（受《测绘法》和测绘数据安全规定约束）？`[待核验]`
> - **重要数据**（DSL 第21条，处理者须承担额外安全保护义务）？
>
> 如果任何为是：行业监管通常提供比 PIPL 更严格的具体要求。通过元典检索最新行业规定并引用。`[元典·法规—待核验]`
>
> 如果没有行业覆盖适用，明确说明——"未识别行业专项监管要求"——以便审查律师看到检查已进行。

## 逐条审查

### 核心条款（检查每个委托处理协议）

逐条审查每个协议。具体立场（通知期、数据位置、安全标准等）来自执业档案 → 个人信息处理协议剧本。PIPL 及配套规定设定的最低要求为监管底线——**在声明底线之前，通过元典核查当前有效规则**。

> **没有静默补充。** 如果元典检索对某制度的要求返回很少或没有结果，报告发现的内容并停止。不要未经询问从训练知识填充空白。
>
> **来源标记分层：**
> - `[元典·法规]`——通过元典工具检索到的法条原文
> - `[元典·案例]`——通过元典工具检索到的案例
> - `[模型知识—待核验]`——训练知识，须核验

| 条款 | 寻找什么 | 剧本字段 | 常见争议 |
|---|---|---|---|
| **角色定义** | 明确委托方为个人信息处理者，受托方为受托处理方；符合实际关系 | — | 将委托处理关系包装为"共同处理"规避PIPL 21条约束 |
| **处理目的和方式** | 严格限于书面约定，不得超出 (PIPL 21条) | — | "及相关目的"等宽泛表述 |
| **处理个人信息种类** | 明确列明，不得超出 | — | 仅列大类，未具体化 |
| **保存期限** | 明确约定，原则上为实现目的所必要的最短时间 (PIPL 19条) | — | "按需保存"等模糊表述 |
| **安全保护措施** | 加密/去标识化/访问控制/等保级别 | Security standards | "采取适当措施"而无具体附件 |
| **转委托权** | 须经委托方书面同意方可转委托 (PIPL 21条第3款) | Subprocessor changes | 设置默认允许转委托，仅需通知 |
| **安全事件通知（受托方通知委托方）** | 通知时限（一般约定24-48小时内）、通知内容 (PIPL 57条参照) | Breach notification | "及时通知"无具体时限；通知范围不清晰 |
| **委托方审计权** | 委托方有权对受托方个人信息处理活动进行监督 (PIPL 21条) | Audit rights | 无审计权条款；或审计程序过于繁琐 |
| **合同终止后数据处理** | 终止后返还或删除，明确期限（如30天），不得保留 (PIPL 21条第2款) | Deletion on termination | "商业上合理"删除；无明确期限；无书面确认义务 |
| **数据位置** | 境内存储（涉及CIIO或规模门槛时）；出境须满足安全评估/标准合同/认证要求 | Data location | 默认允许境外存储无额外限制 |
| **个人权利协作** | 协助委托方响应个人权利请求 (PIPL 59条) | — | 无协助义务条款 |
| **责任承担** | 双方连带责任情形（共同侵害个人权益，PIPL 20条参照）；限额约定 | Liability | 受托方不上限责任 |

### 当我们是受托处理方时：防御性审查

委托方协议试图将运营负担推给我们。对于以下每个条款，与受托处理方剧本比较。

| 条款 | 风险 | 应对 |
|---|---|---|
| 任意转委托须委托方批准（一一审批） | 无法在不逐客户审批情况下调整基础设施供应商 | 应用剧本中转委托的立场；争取采用"提前30天通知 + 无异议视为同意"或"预批准名单"方式 |
| 苛刻的审计条款（随时现场审计无需通知） | 规模上不可行 | 应用剧本中审计权的立场；争取改为独立机构年度审计报告替代 |
| 激进的安全事件通知窗口（如1小时内） | 无法在确认事件前完成通知 | 研究 PIPL 57条+网络数据安全管理条例第11条实践要求；与剧本立场比较；争取"24小时内初次通知+持续更新"机制 |
| 单一国家硬性数据驻留要求 | 可能不匹配技术架构 | 应用剧本中数据位置的立场；确认我们实际可以承诺什么 |
| 受托处理方承担不上限的数据责任 | 经营风险极大 | 应用剧本中责任限制的立场 |
| 委托方可发出任意内容的"指示" | 导致实质上的联合决定处理目的，模糊受托/控制角色 | 将"指示"限定为"协议中明确约定的事项或事先书面同意的具体操作" |
| 极短期限内删除（如合同终止后3天） | 备份轮换和日志保留使此不可能 | 应用剧本中终止后数据处理的立场；约定合理的删除期限（30-90天）和备份/日志豁免 |

### 当我们是个人信息处理者（委托他人）时：保护性审查

供应商协议试图什么都不给我们。对于以下每个条款，与个人信息处理者剧本比较。

| 条款 | 差距 | 应对 |
|---|---|---|
| 无转委托方（分包处理方）披露 | 不知道谁接触我们委托处理的个人信息 | 要求提供当前转委托方清单 + 变更提前通知（至少30天）；涉及个人信息须我方书面批准 |
| "采取适当安全措施"但无具体说明 | 没有实质保障 | 要求附具体安全措施说明（等保级别/加密标准/访问控制机制）或引用命名标准（等保2.0/ISO 27001/SOC 2 Type II） |
| 无安全事件通知时限 | 受托方随时可能晚通知 | 参照 PIPL 57条+网络数据安全管理条例第11条要求约定通知时限（24小时内初次通知） |
| 无审计权 | 无法验证受托方履约情况 | 要求至少每年提交独立审计报告；高风险场景要求我方有权现场审计 |
| 允许用我方数据进行服务改善/分析/AI训练 | 可能在我方数据上训练模型，未经告知将个人信息用于约定范围外目的 | 删除或严格限定；处理仅限于向我方提供约定服务 |
| 无数据出境限制或机制说明 | 可能发生未申报的数据出境 | **通过元典核查适用的数据出境机制**（安全评估/标准合同/认证/例外情形）；要求受托方明确说明数据处理地点和出境合规方式 `[元典·法规—待核验]` |
| 合同终止后无删除/返还承诺 | 数据长期留存于受托方 | 要求约定终止后[N]天内返还或删除，并提供书面确认 |
| 不配合响应个人权利请求 | 个人权利无法得到保障 | 要求受托方配合个人权利请求处理（协助查阅/更正/删除等） |

## 一致性检查：个人信息处理规则

你签署的委托处理协议不能承诺个人信息处理规则（隐私政策）未涵盖的内容，反之亦然。

- 如果协议约定仅出于 X、Y、Z 目的处理——个人信息处理规则是否列出这些目的（PIPL 17条告知义务）？
- 如果处理规则说明某类个人信息仅用于特定目的——协议是否与此一致？
- 如果处理规则列出了特定受托处理方类别——协议受托方是否匹配？

标记不匹配。通常处理规则过时而非协议有误，但需要有人修复其中一个。

## 数据出境条款审查

如果委托处理协议涉及个人信息境外传输，**核查当前有效的数据出境机制要求**（通过元典检索）。对于每个数据流向，识别：

- 适用的出境机制（安全评估/标准合同/认证/例外情形）
- 是否已通过/完成相应程序
- 是否需要更新或重新申报（如安全评估有效期届满）

引用主要来源并验证时效性——数据出境门槛和要求持续演进。标记不确定性以供律师验证。`[元典·法规—待核验]`

如果数据出境机制缺失且存在境外传输，那是 🔴——无合法的出境依据。

## 红线粒度

**以最小可能粒度编辑。** 红线是谈判产物，而非重写。

优先级：替换一个**词** → 替换一个**短语** → 重构一个**子条款** → 替换一个**句子** → 替换**整个条款**（仅在精准编辑比重新起草更难读时）。

有疑问时，越小越好。

> [生成1页摘要：对手方、方向（受托处理方/委托方）、偏离剧本的条款及其解决方式、未决退路决定，以及签署前要问律师的三件事。]

## 此 skill 不做什么

- 它不从零起草个人信息处理协议。如果答案是"使用我们的模板"，从执业档案中的种子文档路径拉取模板。
- 它不做传输影响评估本身——它标记何时需要数据出境合规安排。
- 它不决定是否接受退路之外的条款。它根据升级表路由这些。
