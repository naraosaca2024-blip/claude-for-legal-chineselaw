# /diligence-issue-extraction

1. 加载 `~/.claude/plugins/config/claude-for-legal/corporate-legal/CLAUDE.md` + `~/.claude/plugins/config/claude-for-legal/corporate-legal/deals/[code]/deal-context.md`。
2. 使用以下工作流。
3. 检查 `ai-tool-handoff`——如果类别是批量且工具已配置，首先交接。
4. 阅读文档，应用重要性过滤器，按类别提取。
5. 以内部备忘录格式生成发现。将同意交接给关闭检查清单。

## 事项上下文

**事项上下文。** 检查执业级 CLAUDE.md 中的"## Matter workspaces"。如果 `Enabled` 是 `✗`（内部用户的默认值），跳过本段其余部分——skills 使用执业级上下文，事项机制不可见。如果启用且没有活跃事项，询问："这是哪个事项的？运行 `/corporate-legal:matter-workspace switch <slug>` 或说 `practice-level`。" 为事项特定上下文和覆盖加载活跃事项的 `matter.md`。将输出写入事项文件夹 `~/.claude/plugins/config/claude-for-legal/corporate-legal/matters/<matter-slug>/`。永远不要阅读另一个事项的文件，除非 `Cross-matter context` 是 `on`。

## 目的

VDR 有 2,000 份文档。其中有 30 份对交易很重要。此技能根据 `~/.claude/plugins/config/claude-for-legal/corporate-legal/CLAUDE.md` 中的尽调类别和重要性阈值阅读文档，提取问题，并以内部备忘录格式编写。

## 加载背景

- `~/.claude/plugins/config/claude-for-legal/corporate-legal/CLAUDE.md` → 尽调结构（类别、重要性阈值）
- `~/.claude/plugins/config/claude-for-legal/corporate-legal/CLAUDE.md` → 问题备忘录格式（如何陈述发现）
- `~/.claude/plugins/config/claude-for-legal/corporate-legal/deals/[code]/deal-context.md` → 交易特定阈值、VDR 位置

如果 deal-context.md 不存在，询问这是哪个交易的。

## 工作流

### 步骤 0（前置）：主体穿透核查

**在阅读任何数据室文件之前，先完成主体穿透核查。** 依据法律尽调底盘（GENERAL-005）：不得只停留在合同相对方，须穿透核验以下主体：

**穿透范围：**
- 目标公司（Target Company）
- 转让方（卖方股东）
- 转让方的重要股东（尤其是持有表决权 5% 以上或享有特殊权利的）
- 目标公司的重要子公司/分公司
- 实际控制人

**元典企业工具查询（每个主体执行）：**

```
对每个主体调用以下工具并标注 [元典·企业]：
1. mcp__chineselaw__company_info_by_name — 基本信息（存续状态/注册资本/股权结构/法定代表人）
2. mcp__chineselaw__enterprise_executed_person — 被执行人信息
3. mcp__chineselaw__enterprise_serious_illegal — 严重违法失信名单
4. mcp__chineselaw__enterprise_writ_agg — 司法文书聚合（涉诉情况摘要）
5. mcp__chineselaw__enterprise_executions — 执行信息
```

如元典工具未响应，标注：`元典企业工具未响应 —— [主体名称] 背景未经核查，依赖前请人工核实`

**制裁/失信命中 → 自动升级为 🔴 致命**，路由至交易负责人，说明原因，停止继续审查。

**穿透核查摘要表：**

### 步骤 2：应用重要性过滤器

根据 `~/.claude/plugins/config/claude-for-legal/corporate-legal/CLAUDE.md` / deal-context 阈值。如果阈值说合同 >¥X，则不要审查所有内容。

### 步骤 3：提取问题

对于阅读的每个文档，根据其类别的标准尽调关注点检查：

**重大合同——中国法律框架下的标准提取集：**
- **控制权变更条款**：由本次交易触发？是否需要对方同意或通知？（中国合同中常见"控制权变更"触发条款，但措辞差异大，须逐字核查）`[模型知识—待核验]`
- **转让限制**：合同能否随股权转让而转移给买方？（合同转让须经对方同意，《民法典》第545条）`[模型知识—待核验]`
- **关联交易条款**：合同是否属于目标公司与关联方（股东/董事/高管及其关联方）之间的关联交易？是否履行了内部决议程序？定价是否公允？（公司法第182条）`[模型知识—待核验]`
- **对外担保条款**：目标公司是否提供了对外担保？是否有合规的股东会/董事会决议？（公司法第16条）`[模型知识—待核验]`
- **排他性/竞业禁止**：是否限制买方现有或计划业务？
- **最惠国待遇（MFN）**：是否存在影响定价的最惠国约束？
- **解除权**：对方当事人是否可因本次交易行使合同解除权？
- **违约金/损害赔偿**：是否存在过高违约金风险？（公司法第585条允许法院/仲裁庭调整过高违约金）`[模型知识—待核验]`

**公司（工商/治理）——中国语境下的标准提取集：**
- **工商档案核验**：股权结构与工商登记是否一致？历次股权变更是否有相应决议和工商变更记录？
- **三会文件合规性**：重大决策是否有合规的股东会/董事会/监事会决议？是否符合公司法和章程要求的召集程序和表决门槛？（决议效力6步核验，见 CLAUDE.md `## 中国公司法务方法论`）
- **股东权利矩阵**：优先购买权、共同出售权、领售权、回购权、反稀释权等——是否有相应记录？触发条件是否因本次交易而启动？
- **出资合规性**：注册资本认缴/实缴情况是否如实记录？是否存在出资瑕疵或抽逃出资风险？（公司法2024修订版新规）`[模型知识—待核验]`
- **子公司结构**：重要子公司的治理安排、公司间服务协议、关联方安排
- **反垄断/外资/国资预查**：是否触及经营者集中申报阈值？是否涉及外资准入负面清单？是否涉及国有股权（需要国资主管部门审批）？

**知识产权——中国法律框架下的标准提取集：**
- **权属链**：核心IP（商标/专利/著作权/商业秘密）是否已完成向目标公司的转让登记？创始人/员工是否已依法转让？（职务发明归属，专利法第6条）`[模型知识—待核验]`
- **商标注册状态**：核心商标是否已在相关类别注册？是否有被撤销/无效的风险？
- **软件著作权登记**：核心软件是否已登记著作权？
- **开源软件使用**：产品中的开源组件是否符合许可协议要求？
- **知识产权纠纷**：是否有未决或威胁中的知识产权诉讼/无效宣告程序？
- **许可vs自有**：核心IP是许可使用还是完全拥有？许可是否可随交易转移？

**劳动用工——中国法律框架下的标准提取集：**
- **劳动合同合规性**：是否签订书面劳动合同？是否存在未签合同的双倍工资风险？（劳动合同法第82条）`[模型知识—待核验]`
- **社保公积金合规性**：是否为全体员工缴纳社保公积金？欠缴金额估算？（这是中国企业最常见的劳动合规红旗之一）
- **竞业限制协议**：核心员工是否签署竞业限制协议？是否按月支付竞业限制补偿金？（竞业限制最长2年，劳动合同法第24条）`[模型知识—待核验]`
- **控制权变更安排**：是否存在因控制权变更触发的补偿条款？（中国法律通常无"黄金降落伞"，但可约定）
- **关键员工留任风险**：核心员工的劳动合同剩余期限？是否存在留任关键人员的约束？
- **劳务派遣/外包风险**：是否存在大量劳务派遣工或外包员工？是否符合劳务派遣相关规定？
- **未决劳动争议**：是否有在仲裁或诉讼中的劳动争议？

**诉讼仲裁——中国法律框架下的标准提取集：**
- **未决民事诉讼**：未决诉讼事项、争议金额、预计结果
- **未决仲裁**：是否有在仲裁委员会进行的仲裁程序？
- **行政处罚**：是否有工商/税务/环保/市场监管等行政机关的行政处罚记录？
- **监管询问/调查**：是否正在接受任何监管机构的调查？
- **已知威胁索赔**：是否有合理预期会引发诉讼或索赔的风险？

**税务——中国法律框架下的标准提取集：**
- **税务合规性**：是否按时申报并缴纳各项税款？是否有税款欠缴？
- **税收优惠资质**：高新技术企业/小微企业等税收优惠资质是否合规取得和维持？
- **关联交易定价**：与关联方的交易定价是否符合独立交易原则？是否面临转让定价调查风险？
- **股权转让涉税**：本次交易的股权转让是否涉及个人所得税/企业所得税的申报义务？

**资质许可——中国法律框架下的标准提取集：**
- **经营许可/资质**：目标公司的核心业务是否有所需的行政许可和资质？是否存在无证经营风险？
- **特许行业合规性**：是否涉及金融/电信/教育/医疗/广播电视等特许行业？相关许可是否齐全有效？
- **数据合规资质**：是否涉及个人信息处理、数据出境等需要合规证明的事项？（PIPL/CSL/DSL合规）

### 步骤 4：陈述每个发现

> **来源归因。** 当发现引用法规、条例、案例或监管行动时，用引用来源标记引用：`[元典·法规]`、`[元典·案例]`、`[元典·企业]`（用于从元典工具检索的引用）；`[联网检索—需复核]`（用于网络搜索引用）；`[模型知识—待核验]`（用于从训练数据回忆的引用）；`[user provided]`（用于来自数据室、交易团队备忘录或外部律师反馈的引用）。文档来源引用（数据室路径、文件名）保留其原始参考。标记为 `待核验` 的引用具有更高的编造风险，应优先检查。永远不要删除或折叠标记。
>
> **当不同意用户引用的法规时，引用文本或拒绝描述它。** 如果用户（或交易团队备注，或卖方披露）引用法规来支持你认为不正确的主张，并且你没有从连接的研究工具或 VDR 获得法规文本可用，不要编造法规内容的描述。而是说："那个部分不符合我对 [批量销售通知 / 继任责任 / 无论什么] 要求的期望——我需要提取实际文本才能告诉你它实际涵盖的内容。`[statute unretrieved — verify]`"然后要么 (a) 通过配置的研究工具检索文本并引用它，(b) 要求用户粘贴文本，或 (c) 标记为外部律师。对真实法规的自信错误描述比"我不知道"更糟糕——引用编造子章节的交易团队备忘录比差距更难让人不再相信。适用于描述法规的每个 skill，而不仅仅是问题提取。
>
> **没有静默补充。** 如果对配置的法律研究工具的研究查询对于发现所需的法律依据返回很少或没有结果（例如，控制权变更同意要求的规则、IP 转让原则、雇佣分类测试），报告发现的内容并停止。不要在未询问的情况下从网络搜索或模型知识填充差距。说："搜索从 [tool] 返回了 [N] 个结果。对于 [规则 / 原则]，覆盖范围似乎很薄。选项：(1) 扩大搜索查询，(2) 尝试不同的研究工具，(3) 搜索网络——结果将标记为 `[web search — verify]`，在依赖前应根据主要来源检查，或 (4) 标记为未验证并停止。你想要哪个？"律师决定是否接受较低可信度的来源。

根据 `~/.claude/plugins/config/claude-for-legal/corporate-legal/CLAUDE.md` 中的发现模板。如果种子备忘录使用了这个：

```
Issue #N: [标题]
Category: [请求清单类别]
Severity: [根据内部方案的级别]
Documents: [VDR 路径 + 文档名称]
Finding: [文档说什么以及为什么重要]
Recommendation: [价格调整 / 赔偿 / 需要同意 / 陈述与保证 / 放弃]
```

……然后完全使用那个。如果种子备忘录是项目符号，写项目符号。

**严重性校准（中国法律尽调标准四级体系）：**

| 等级 | 含义 | 处置建议 |
|---|---|---|
| 🔴 **致命/阻断** | 交易障碍，不解决不能闭环。典型：无法弥补的权属瑕疵、重大合规违法、主体穿透发现制裁/失信 | 治愈方案+时间表；不可治愈则中止建议 |
| 🟠 **重大/高** | 重要风险，需在交易文件中设置专项保护。典型：未缴社保/税款欠缴估算清楚但可补缴、待解除的控制权变更同意 | 陈述保证覆盖+赔偿安排+补救期+托管 |
| 🟡 **一般/中** | 一般风险，知会即可，在报告中披露。典型：轻微程序瑕疵、可修复的格式缺陷 | 报告披露+监控 |
| 🟢 **提示/低** | 提示性事项，备查不影响交易。典型：合规建议事项、规范化建议 | 备查 |

**每项风险必须回答：**
1. 是否影响交易（能否继续交割）？
2. 影响哪些具体条款（陈述保证/赔偿/先决条件/价款调整）？
3. 可否补救？谁承担补救成本？
4. 需要补充什么材料？

**红旗→交易条款映射（每个 🔴/🟠 发现必须）：**

| 发现 | 处理选项 | 交易条款映射 |
|---|---|---|
| [具体风险] | [补充披露/整改/价款调整/赔偿承诺/先决条件/终止] | [SPA第X条/陈述保证第X条/赔偿第X条] |

## Handoffs

- **交给 ai-tool-handoff：** 如果按 CLAUDE.md 配置了批量合同审查工具（元典或其他国内AI合同审查工具），将批量合同交接。此 skill 处理需要判断层的文件（附函、补充协议、AI工具难以处理的内容）。
- **To deal-team-summary:** Aggregated findings feed the deal team brief.
- **交给 material-contract-schedule：** 合同级别的提取输入披露清单。
- **交给 closing-checklist：** 任何暗示离散关闭前行动的发现都会成为检查清单项目。交接不仅限于第三方同意——它还涵盖：
  - **股东投票 / 其他关闭行动**——§280G 清洗投票、要求的股东同意、要求的董事会决议、评估权通知期、转换机制，或交易完成所需的任何其他公司批准。表征行动、批准阈值、法定或章程来源以及时间约束。
  - **监管申报和批准**——经营者集中反垄断申报（商务部）、外资准入审查（发改委/商务部）、国资审批（国资委）、证监会核准/注册（上市公司）、行业许可机构的专项审批，以及提取期间标记的其他行业特定批准。
  - **来自对手方的同意**——控制权变更、反转让、MFN 触发同意。
  - **解除、终止或清偿**——与控制权变更相关的雇佣解除、清偿信、留置权解除。
  - **托管 / 扣留机制**——如果提取 surfaced 赔偿托管、R&W 保险交付物或与特定问题相关的扣留。
  每个带有关闭前行动标签的发现都应该到达 closing-checklist，而不仅仅是标记为"同意"的那些。如果发现处于灰色区域（可能需要关闭行动，可能是关闭后承诺），标记并交接——如果购买协议另有说明，closing-checklist 可以放弃它。交接不足是单向门；交接过度在审查中纠正。


**继任责任。** 标记：未决或威胁的侵权/产品责任索赔、环境问题和清理义务、批量销售/欺诈转让敞口（卖方是否保留足够资产支付其余债权人？）、卖方关闭后解散计划（如果卖方解散，原告会追逐买方），以及购买协议是否有实际覆盖已知敞口的假设/除外责任清单。即使在资产交易中，"事实合并"、"仅仅是延续"和"产品线"原则也可以转移责任——这是让认为他们正在干净购买资产的买方客户感到惊讶的分析。

## 批量处理

对于大型类别（300 份合同），分批处理。每批之后，更新运行中的问题列表并立即标记任何 🔴 项目——不要等待整个类别来 surfaced 影响交易的问题。

## 以下一步决策树结束

根据 CLAUDE.md `## Outputs` 的下一步决策树结束。自定义此技能刚生成的内容的选项——五个默认分支（起草 X、升级、获取更多事实、观察等待、其他）是起点，而不是锁定。树就是输出；律师来选择。

如果提取 surfaced 超过 ~10 个问题，或任何时候用户询问：提供仪表板（请参阅 CLAUDE.md `## Outputs → Dashboard offer for data-heavy outputs`）。为此输出塑造提议——按严重性计数（🔴 / 🟠 / 🟡 / 🟢）、按内部类别计数，以及带有重要性、类别和 VDR 来源的可排序问题网格。

## 此技能不做什么

- 它不对接近的案例做出重要性判断。它应用阈值；人类决定边界。
- 它不谈判陈述和保证。它生成告知它们的发现。
- 它不替代批量 AI 审查。对于大批量合同条款提取，根据 CLAUDE.md 交接给配置的批量审查工具（元典或其他工具）。此 skill 用于判断层。
- 它不替代财务/税务顾问。对财务材料的法律使用，只做法律相关性判断；财务数据真伪和计量由会计师判断。
- 它不对元典企业查询结果作为最终核实结论——元典是辅助工具，重要主体的工商档案须通过官方渠道（国家企业信用信息公示系统/企查查/天眼查）原件核验。
