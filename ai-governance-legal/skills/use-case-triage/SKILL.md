# /use-case-triage（AI 用例合规分类）

1. 阅读执业档案。确认用例登记册和红线已填充——如果没有，停止并指向设置。
2. 使用以下框架。如果模糊则阐明用例。
3. 注册表查找 → 红线检查 → 法规覆盖分析 → 分类。
4. 输出：分类结论、推理、条件表（如有条件）、治理层级、跨 plugin 交接。
5. 如果用例尚未在注册表中，提议注册表更新。

```
/ai-governance-legal:use-case-triage "销售团队想用AI自动对线索进行评分"
```

## 目的

中国 AI 监管以**法规覆盖**为判断核心：一个用例可能同时触发多部法规，每部法规有独立义务。此 skill 系统地检查每部适用法规，给出三色结论：
- 🟢 **已批准**：无需额外措施（或仅需常规合规）
- 🟡 **有条件审批**：须满足特定条件（备案/安全评估/内容标识等）方可推进
- 🔴 **不批准**：触碰红线，不得推进

## 加载上下文

读取执业档案：
- `## 用例登记册` — 查找相同或类似用例
- `## 红线` — 一票否决清单
- `## 监管足迹` — 公司已适用的法规
- `## 供应商 AI 立场` — 如涉及第三方AI

## 步骤 1：理解用例

如用例描述模糊，先提出最多 3 个澄清问题再进行分析：

1. 谁是最终用户（内部员工/外部客户/境内公众）？
2. AI 如何介入决策过程（辅助推荐/主导决策/全自动执行）？
3. 处理哪些数据（个人信息/人脸/声纹/行为数据/重要数据）？

---

## 步骤 2：注册表查找

在执业档案 `## 用例登记册` 中查找是否有相同或类似用例，已有结论时直接引用并询问是否有变化。

---

## 步骤 3：红线检查（一票否决）

逐项检查以下红线。**触碰任何一项，立即输出 🔴 不批准，不再进入步骤 4。**

| 红线 | 法律依据 |
|---|---|
| 生成或传播违反社会主义核心价值观、危害国家安全的内容 | 《生成式AI办法》第4条；《算法推荐管理规定》第14条 |
| 基于用户个人特征（支付能力/消费习惯）实施差别定价 | 《算法推荐管理规定》第21条 |
| 在宾馆客房、公共浴室、更衣室、公共卫生间安装人脸识别设备 | 《人脸识别办法》第13条 |
| 将人脸识别设为门禁等场景的唯一身份验证方式（无替代方案） | 《人脸识别办法》第10条；最高法司法解释第10条 |
| 向未成年人提供虚拟亲属/虚拟伴侣等虚拟亲密关系服务 | 《拟人化互动办法》第14条（2026-07-15施行）`[法规尚未施行]` |
| 以捆绑授权/强迫方式要求用户同意处理人脸信息 | 《人脸识别司法解释》第4条 |
| 智能体自主决策超出用户授权范围 | 《智能体实施意见》第6条 |
| 将替代社会交往、控制用户心理、诱导沉迷依赖作为拟人化互动服务目标 | 《拟人化互动办法》第10条（2026-07-15施行）`[法规尚未施行]` |
| 执业档案中列出的公司自定义红线 | [见执业档案 ## 红线] |

## 步骤 4：中国 AI 法规覆盖分析

逐一检查以下法规是否适用于该用例。**如元典已连接，优先通过元典核验适用条款。**

### 4a. 生成式AI（《生成式人工智能服务管理暂行办法》，2023-08-15）

触发条件：利用生成式AI技术向**境内公众**提供服务（文本/图片/音频/视频/代码等生成）。

如适用，检查：
| 义务 | 要求 | 现状 |
|---|---|---|
| 安全评估 | 上线前须通过安全评估 | [已完成/未完成/不适用] |
| 备案 | 安全评估通过后向国家网信办备案 | [已备案/未备案/不适用] |
| 内容标识 | 生成内容须标注AI标识 | [已实现/未实现] |
| 拦截机制 | 须能识别并拒绝违规请求 | [已实现/未实现] |
| 训练数据合规 | 训练数据须合法，不侵犯知识产权 | [待评估] |
| 用户协议 | 须与用户签订服务协议 | [已有/待建立] |

`[元典·法规—待核验最新备案要求]`

### 4b. 算法推荐（《互联网信息服务算法推荐管理规定》，2022-03-01）

触发条件：利用算法向用户推荐信息、内容、商品、服务，**且具有舆论属性或社会动员能力**（向公众提供且用户规模达到一定量级）。

如适用，检查：
| 义务 | 要求 |
|---|---|
| 算法备案 | 须向国家互联网信息办公室备案（第24条） |
| 用户标签告知 | 须提供查看个人特征标签的渠道（第17条） |
| 拒绝推荐权 | 须提供关闭个性化推荐的选项（第17条） |
| 禁止行为 | 不得实施差别定价/流量造假/捆绑绑架/算法压榨（第20-21条） |

`[模型知识—待核验当前备案阈值和流程]`

### 4c. 深度合成（《互联网信息服务深度合成管理规定》，2023-01-10）

触发条件：提供深度合成服务——利用深度学习/虚拟现实技术合成文本/图像/音频/视频/虚拟场景（第2条）。

如适用，检查：
| 义务 | 要求 |
|---|---|
| 显著标识 | 合成内容须显著标识（第16条） |
| 真实身份核验 | 须核验用户真实身份（第9条） |
| 备案（具舆论属性的） | 具有舆论属性的深度合成服务须备案（第19条） |
| 人脸人声禁止滥用 | 不得利用他人人脸/声音制作虚假信息（第14条） |

`[元典·法规—待核验]`

### 4d. 人脸识别（《人脸识别技术应用安全管理办法》，2025-06-01）

触发条件：利用人脸识别技术处理人脸信息（采集/存储/比对/辨识）。

如适用，检查：
| 义务 | 要求 |
|---|---|
| 必要性评估 | 须论证人脸识别的必要性（第7条） |
| 单独同意 | 须取得个人单独同意（第6条） |
| 提供替代方案 | 不得将人脸识别设为唯一方式（第10条） |
| 场所限制 | 公共场所须经批准且显著标识（第12条） |
| 技术应用备案 | 须向省级主管部门申请备案（第18条）`[待核验备案要求]` |
| 未成年人保护 | 处理未成年人人脸信息须取得监护人同意（第8条） |

`[已确认《人脸识别办法》施行日期 2025-06-01—最后核验 2026-05-31]`

### 4e. AI内容标识（《人工智能生成合成内容标识办法》，2025-09-01）

触发条件：向社会公众提供利用AI生成合成的内容。

核心义务：须对AI生成内容进行显著标识；技术标准参见 GB/T 45438-2025。`[元典·法规—待核验]`

### 4f. 智能体（《智能体规范应用与创新发展实施意见》，2026-05-09）

触发条件：开发或部署具备自主感知、记忆、决策、交互与执行能力的智能体系统。

核心义务：
- 确保智能体不超出用户授权范围自主决策（第6条）
- 敏感领域须备案/检测（第11条）`[法规刚颁布，具体备案要求待细化]`

### 4g. 拟人化互动服务（《拟人化互动办法》，2026-07-15施行，**尚未施行**）

触发条件：提供模拟自然人人格特征的**持续性情感互动服务**（情感照护/陪伴/支持）。
不适用：智能客服/知识问答/工作助手/学习教育/科研（非持续性情感互动）。

如适用，核心义务（施行后）：
- AI身份标识（须明示正与AI互动，第18条）
- 安全评估并向省级网信部门提交评估报告（第22条）
- 未成年人保护（禁止向未成年人提供虚拟亲密关系，第14条）
- 过度依赖防范（连续2小时须提醒，第18条）
- 数据限制（敏感个人信息交互数据未经单独同意不得用于模型训练，第16条）

`[法规尚未施行—2026-07-15施行；当前可预判义务但不作强制要求]`

## 此 skill 不做什么

- 它不对接近的案例做出重要性判断。它应用法规框架；人类决定边界。
- 它不替代完整的 AIA（AI合规评估）。有条件批准的高风险用例应运行 `/ai-governance-legal:aia-generation`。
- 它不涵盖中国内地以外的 AI 治理法规（EU AI Act/GDPR等）。
