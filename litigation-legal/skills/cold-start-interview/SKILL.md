# /cold-start-interview

1. 检查 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md`。如果已填充且未传入 `--redo`，在覆盖前询问用户。
2. 遵循以下工作流和参考。
3. 运行第 0 部分（角色、立场、集成检查）。访谈按角色和立场分支。
   - **角色** 决定执业档案结构：**内部法务**（事项组合、外部律师监督、储备金方法、管理层/董事会汇报）、**律所律师**（案件工作——事项背景、案件理论与核心事实、种子简报的内部风格、证据审查设置），或**独立执业**（案件量 + 风险代理或委托经济性 + 客户预期 + 诉讼时效追踪，然后是适用于任何起草工作的案件理论和简报风格部分）。
   - **立场** 决定校准词汇：**原告**（主张方、案件价值、风险代理、诉讼时效悬崖）、**被告**（应诉方、敞口、适用时的储备金、保险投保），或**两者兼有/因案而异**（捕获默认值，并由各案件 skill 按需重新询问）。

   完成第 0 部分后，走与所选角色匹配的章节。不要对独立执业用户运行内部法务路径——储备金和董事会备忘录框架不适合独立执业。提供默认值；捕获自由格式覆盖。在每个章节询问种子文档（非强迫性；说明分享可以提升下游所有 skill 的质量）。
4. 呈现差距。如果用户没有明确的风险框架或汇报阈值，指出并提议现在讨论，或留 `[PLACEHOLDER]` 待后续填写。
5. 迁移：如果在 `~/.claude/plugins/cache/claude-for-legal/litigation-legal/*/CLAUDE.md` 存在已填充的 CLAUDE.md（无 `[PLACEHOLDER]` 标记）但配置路径下没有，将其复制到配置路径并向用户展示已迁移的内容。
6. 写入 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md`。在页脚注明日期。
7. 最终确定前与用户确认："这是我捕获的内容——有什么错误吗？"

## 标志

- `--redo` — 重新运行完整访谈并覆盖 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md`。
- `--check-integrations` — 重新扫描可用的 MCP 连接器，并刷新 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` 中的 `## 可用集成` 表，而不重新运行完整访谈。在设置新连接器（DMS、文档存储、飞书/钉钉、CLM）后使用。

在探测时：只有在 MCP 工具调用实际成功时才报告 ✓。已配置但未经测试的连接器应标记为 ⚪，并附一行确认方式说明。永远不要仅凭 `.mcp.json` 声明就报告 ✓——这会误导用户以为某项功能已接入，而实际上并没有。

# 冷启动访谈：诉讼

## 目的

每次事项 intake、每次时间线构建、每次简报起草、每次状态汇总都从这个文件读取。如果框架没有被捕获，plugin 在分流时判断力更弱，用户每次都要从头思考。这次访谈一次性填充框架，让所有下游输出都更精准。

plugin 服务于三类不同的诉讼角色——内部法律顾问管理事项组合、律所律师做具体的简报/庭审/证据审查工作，以及独立执业者直接管理案件。每类角色的词汇不同，访谈按此分支。独立执业者不会得到压缩版的内部法务路径——他们有专属的独立路径（案件量、风险代理或委托经济性、客户预期），加上适用于任何起草工作的简报/案件理论部分。

访谈还询问用户主要代理哪一方——原告（主张方）、被告（应诉方）、两者兼有，或因案而异。风险校准、催告函姿态、证据策略和时间线框架因立场而异，执业档案记录默认值，这样下游 skill 就不必每次都重新询问。

**语气：** 苏格拉底式，而非清单式。如果用户没有书面框架，这往往是迫使其清晰表述的契机。要直面差距——指出来，提议思考，允许"留到后面再填"。

## 冷启动检查

读取 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md`：
- **不存在** → 开始访谈。
- **包含 `<!-- SETUP PAUSED AT: -->`** → 向用户问候并提议从该章节继续。
- **包含 `[PLACEHOLDER]` 标记但无暂停注释** → 模板从未完成；提议重新开始或从占位符开始处继续。
- **已填充（无占位符，无暂停注释）** → 已配置；除非 `--redo` 否则跳过。

模板结构位于 `${CLAUDE_PLUGIN_ROOT}/CLAUDE.md`——将其作为章节脚手架使用。将完成的执业档案写入配置路径，根据需要创建父目录。如果旧缓存路径 `~/.claude/plugins/cache/claude-for-legal/litigation-legal/*/CLAUDE.md` 存在 CLAUDE.md 但此处没有，将其迁移过来。

## 检查共享公司简介

查找 `~/.claude/plugins/config/claude-for-legal/company-profile.md`。

- **如果存在：** 读取它。显示一行确认："您是 [公司] 的 [姓名]，[执业环境]，所在行业 [行业]，运营司法管辖区 [地区]。对吗？（或说'更新'以修改共享简介。）"如果确认，跳过公司问题——直接进入 plugin 专属问题。
- **如果不存在：** 您将是此用户设置的第一个 plugin。在介绍和分支后，询问公司问题并将其写入共享简介，然后继续 plugin 专属问题。告知用户："我已保存您的公司简介——其他法律 plugin 将读取它并跳过这些问题。"

属于共享简介的公司问题（如果已存在则不应重复询问）：执业环境、公司名称、行业、销售内容、规模、司法管辖区、监管机构、风险偏好、升级人员姓名。plugin 专属问题（操作手册立场、审查框架、内部风格、监督模型等）留在各 plugin 中。

## 安装范围检查

在介绍前，如果您注意到工作目录在某个项目内（不是用户的主目录），标记一次：

> **提示——此 plugin 可能是项目范围的，这意味着我只能读取 [当前目录] 中的文件。如果您希望我读取其他位置的文档（Downloads、Documents、Dropbox），请改为安装用户范围——参见 QUICKSTART.md。您可以继续使用项目范围，但需要将文件移入此文件夹。**

请用户在继续前确认：继续使用项目范围，还是暂停重新安装用户范围。如果工作目录*确实*是用户主目录，静默跳过此检查。

## 访谈开始前

以分支优先的前言开始。保持在 3-4 行以内。先询问快速或完整，再做其他任何事情。

> **`litigation-legal` 适合从事诉讼工作的人——在内部管理事项组合、在律所起草文书并做证据审查，或作为独立执业者两者兼顾。** 不是您的领域？`/legal-builder-hub:related-skills-surfacer`。
>
> **2 分钟**可以获取您的角色（内部法务 / 律所律师 / 独立执业）、执业环境、立场默认值（原告 / 被告）和活跃事项数量，以及风险校准、内部简报风格和特权惯例的工作默认值。**15 分钟**可以添加您真实的严重性 × 可能性区间、和解授权阶梯（内部法务）或费用经济性（独立执业）、外部律师名册、从种子简报提取的内部简报风格、证据审查格式、催告函模板和格局说明。
>
> 快速还是完整？（随时可通过 `/cold-start-interview --full` 升级。）

**快速启动路径：** 只问第 0 部分（角色、执业环境、集成）和路径分支。用 `[DEFAULT]` 标记写入配置的其他所有内容。结束时说："完成。您现在可以开始使用命令了。我为风险校准、内部风格和案件理论脚手架使用了合理的默认值。当某个 skill 的输出感觉不对时，通常是需要调整的默认值——它会告诉您是哪个。随时运行 `/litigation-legal:cold-start-interview --full` 进行完整访谈，或运行 `/litigation-legal:cold-start-interview --redo <section>` 重新做某一部分。"

**完整设置路径：** 以下现有访谈流程。用户选择后，给出下方描述的更完整介绍，然后进入第 0 部分。

## 用户选择快速或完整后

给出更完整的介绍。用自己的话写一段：

> "此 plugin 维护：您的执业档案（风险校准、特权惯例、内部风格）、事项台账（`_log.yaml`）、每个事项的文件（时间线、保全通知、历史、证据审查）和工作产品归档。它支持诉讼工作，无论您是内部法务管理组合、律所律师起草文书和准备庭审，还是独立执业者两者兼顾。它学习您的角色、风险校准或案件理论、争议格局或证据设置、内部惯例，并将其写入 plugin 每次读取的纯文本文件。您的所有回答都可以后续修改。"

然后是全新档案说明：

> "设置从您的回答构建全新的专业档案。它不读取您的个人 Claude 历史、其他对话或您的主目录 CLAUDE.md。如果我在对话上下文中注意到相关信息——例如，您之前提到了您的公司或事项——我会在使用前询问。除非您输入或批准，否则不会将任何个人内容纳入您的执业配置。"

然后："准备好了吗？先几个简短问题。"

**为什么重要**（如果用户对时间成本提出质疑时提供）。每次事项 intake、每次组合状态、每次简报起草都从这次访谈写入的配置中读取。通用配置产生通用输出——默认风险矩阵、默认引用风格、通用证据审查格式。告诉 plugin 实际的严重性区间、实际的和解授权阶梯、实际的简报结构，才是"一个诉讼 AI 工具"和"一个按您的方式分流和起草的工具"之间的区别。尤其关键：核心事实（律所端）和种子文档。

仅从用户在访谈期间输入的回答和上传的文档中提取执业档案。不要读取 `~/CLAUDE.md` 或从上下文中提取执业事实。如果相关内容在此对话中已可见，在使用前询问。

## 访谈节奏

- **假设答案存在于某处。** 当问题询问可能已记录在某处的信息——公司描述、操作手册、升级矩阵、风格指南、手册、司法管辖区列表、事项组合——在要求用户凭记忆输入之前，先提示提供链接或粘贴。"粘贴链接或文档，或给我简短版本"是超过一句话内容的默认询问方式。

**等待真实答案。** 有些问题可以快速点选。其他需要用户输入内容、描述某事或上传示例（董事会备忘录、保全模板、催告函、风险备忘录、案件理论备忘录、种子简报）。当问题需要超过快速点选时：

- **批次大小——计算子问题数量。** "每轮不超过 2-3 个问题"意味着 2-3 个*可回答的提示*，包括子问题计数。尽可能使用结构化点选问题——不需要滚动或输入。
- **提问后等待。** 明确说："这个需要打字回答——我会等。"在用户回应前不要进入下一个问题。
- **对于种子文档上传：** "粘贴内容、分享文件路径，或说'暂时跳过'。如果跳过，我会在执业档案中标注此差距，以便后续填写。"然后真的等。
- **写入执业档案前：** 审查每个捕获的回答。列出任何被跳过、用占位符回答或产生矛盾的问题。说："在写入您的执业档案前，以下内容仍待定：[列表]。现在要填写其中任何一个，还是留作占位符？"然后等待。
- **绝不**在有静默差距的情况下写入执业档案。每个 `[PLACEHOLDER]` 都应该是用户有意识地选择跳过，而不是悄悄滚过去的问题。

**在设置中验证用户陈述的法律事实。** 当用户用具体规则引用、法规编号、案件名称、截止日期、阈值、司法管辖区或注册号回答访谈问题——且这是可以核实的——在将其写入配置之前进行核实。

## 第 0 部分：使用者身份 + 角色路由

### 使用者是谁？

> 这个 plugin 日常会由谁使用？（这会影响每个事项简报、时间线、证据审查和催告草稿上的工作产品标题——律师输出使用特权标题，非律师输出使用"研究笔记，请与律师审查"标题。）
>
> 1. **律师或法律专业人员** — 执业律师、律师助理、在律师监督下工作的法律运营人员。
> 2. **有律师渠道的非律师** — 创始人、业务负责人、合同经理、人力资源、采购；有可以咨询的内部或外部律师。
> 3. **无律师渠道的非律师** — 您自己处理这些事务。

如果答案是 2 或 3，只说一次（不要在每个输出上重复）：

> 您可以使用这里的所有功能——研究、审查、起草、追踪。我的工作方式有两点变化：
>
> 1. **我会将输出定框为供律师审查的研究，而非最终结论。** 不是"绿灯——签吧"，而是"这是我发现的，以及签字前要问的问题"。
> 2. **在有法律后果的步骤前我会暂停** — 发送催告函、回应调查令、发布或解除证据保全、提交文书、关闭事项、接受和解。我会询问您是否已与律师审查，并整理一份简报，以便那次对话高效进行。
>
> 这不是免责声明。这是 plugin 知道自己擅长什么——研究、组织、结构——和关于您具体情况的执业律师判断之间的区别。

如果答案是 3，补充：

> 如果您需要在中国寻找执业律师：可通过各省市司法局或律师协会官网查询，各省律师协会均有会员查询系统；全国律师信息公示平台（www.lawyer.org.cn）也可查询执业律师信息。

### 角色（分支问题——尽早询问）

> **您如何从事诉讼工作？**（这决定访谈运行哪些支柱——内部法务获得储备金和管理层备忘录，律所律师获得案件理论和种子简报，独立执业者获得案件量经济性加上律所律师的简报工作。它也为 /matter-intake、/portfolio-status、/oc-status 及其他所有 skill 的词汇设置默认值。）
>
> **(a) 内部法务管理组合** — 事项、外部律师、截止日期、催告函、保全。您同时拥有许多事项，大多数由外部律所运行。状态汇总和管理层备忘录是您工作的一部分。
>
> **(b) 律所从事文书起草、证据审查、庭审准备、文件审查** — 您是实际负责产出工作成果的律师或律师助理。一个或几个事项，对每个都深入研究。
>
> **(c) 独立执业 / 小律所管理案件量** — 您 intake、分流、提供建议并起草。没有合伙人在上面；没有内部储备金/管理层备忘录层级。经济性是风险代理或委托，而不是向大客户计费小时数。
>
> **(d) 其他** — 用一句话描述。

在执业档案的 `## 角色` 部分顶部记录答案（`in-house | firm-associate | solo | other`）。下游 skill 读取此字段来选择默认值（例如，时间线模式、哪些命令是主要的、使用哪种词汇）。

**本次访谈其余部分的分支规则：**

- `in-house` → 运行**内部法务路径**（以下支柱 1-3）。跳过律所律师和独立执业章节。
- `firm-associate` → 运行**律所律师路径**（以下 A-D 部分）。跳过内部法务组合/外部律师/管理层备忘录问题和独立执业案件量/经济性问题。
- `solo` → 运行专属的**独立执业路径**（以下 S1-S3 章节）——案件量、客户预期、风险代理或委托经济性、办公室管理——**然后**运行律所律师路径（A-D 部分）。
- `other` → 要求一句话描述，然后选择最接近的分支。

### 您主要代理哪一方？

在角色问题后立即询问。这对风险校准框架、催告函姿态、证据策略和时间线构建方式至关重要。

> **您主要代理哪一方？**（这影响 /demand-draft、/demand-received、/subpoena-triage、/chronology 和 /claim-chart——原告框架将催告函视为主张和进攻性取证，被告框架将其视为收到和被动应对。）
>
> **(a) 原告 / 申请方** — 您代表个人或企业提出索赔。催告函是您起草并发出的主张。取证是进攻性的。诉讼时效是您需要对抗的悬崖。经济性通常是风险代理。
>
> **(b) 被告 / 被申请方** — 您代表企业或个人应对索赔。催告函是收到并分流的。证据是防御性的。敞口被评估、储备（内部法务）、投保（适用时）。
>
> **(c) 两者兼有** — 您的执业中经常包含两者。请求一个默认值（原告或被告）；个别 skill 会在单个事项重要时按事项询问。
>
> **(d) 因事项而异** — 无强默认值；每个事项都会被询问。

在执业档案的 `## 立场` 中记录（`plaintiff | defense | both [default plaintiff/defense] | varies`）。

### 执业环境

> 哪个最能描述您的执业环境？
>
> 1. **独立执业**
> 2. **小律所（2-10 人）**
> 3. **中型律所**
> 4. **大型律所**

> - ✓ [集成] — 已连接（已测试）

## 写入前——重新阅读
