# /chronology

1. 加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/[slug]/matter.md` → 理论、枢纽事实、关键事实。
2. 加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` → 文档存储来源、默认事项文件夹模式。
3. 遵循以下工作流和参考。
4. 按顺序识别来源：本次会话中用户提供的路径、默认事项文件夹、从 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` 声明的来源。
5. 对于可读来源：提取日期事件。对于无法访问的来源：在差距中注明。
6. 去重，根据事件与来源列表合并。
7. 根据事项理论标记重要性 (🔴/🟡/⚪)。
8. 编写 `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/[slug]/chronology.md`（或根据标记的格式变体）。
9. 如果存在先前版本：版本号递增，向用户展示差异摘要。
10. 在完成之前确认："这是我构建的内容。扫描 🔴 条目——我误分类了什么吗？"

# 年表/时间线

## 证据材料使用限制

在处理诉讼证据材料之前，询问："这些材料中是否有任何一份是通过法律程序中的调查令、法院查证或证据交换获得的？"如果是：

- **中国民事诉讼：** 依《民事诉讼法》第68条及《最高人民法院关于民事诉讼证据的若干规定》，当事人在诉讼活动中知悉的对方当事人的商业秘密，应当予以保密，不得在本案以外使用。通过法定程序获取的证据材料，应仅用于本案诉讼目的。
- **仲裁程序：** 依各仲裁机构规则（CIETAC、上仲等），通过仲裁程序取得的证据材料通常负有保密义务，使用范围须符合仲裁协议约定。
- **行政程序：** 通过行政复议或政府信息公开获取的材料，使用目的须与申请目的一致。

确认："此使用在本案诉讼或仲裁程序范围内，且合规。"如果未确认，请标记："⚠️ 部分证据材料可能有使用限制。在继续之前确认此使用是否允许。"

## 目的

事实按顺序发生。时间线是每个叙述所依赖的脊柱——文书中的事实陈述、保全备忘录、和解备忘录、庭审准备、证人准备。手工构建时间线很慢；AI 擅长于结构化提取。问题：垃圾进，垃圾出。此 skill 从配置声明的来源和用户上传的任何内容中提取。

## 中国诉讼关键时间节点

在构建时间线时，重点标记以下中国诉讼程序的关键时间节点：

| 时间节点 | 重要性 | 说明 |
|---|---|---|
| 合同订立 | 🔴 | 权利义务起算点 |
| 合同履行开始 | 🟡 | 义务履行期间 |
| 催告/通知 | 🔴 | 诉讼时效中断事件（民法典第194-196条） |
| 违约/违约发现 | 🔴 | 诉讼时效起算点 |
| 合同解除 | 🔴 | 解除权行使时间 |
| 诉前保全申请 | 🔴 | 财产保全时间（民诉法第100-105条） |
| 证据保全申请 | 🔴 | 证据保全时间（民诉法第81条） |
| 立案日期 | 🔴 | 诉讼开始时间 |
| 答辩期届满 | 🟡 | 被告答辩期（15天） |
| 举证期限届满 | 🔴 | 证据提交截止 |
| 庭前会议 | 🟡 | 庭前证据交换（民诉法第133条） |
| 开庭日期 | 🔴 | 庭审时间 |
| 裁判/判决 | 🔴 | 裁判结果时间 |
| 上诉期届满 | 🟡 | 一审判决生效时间（15天）|
| 上诉提起 | 🟡 | 二审启动时间 |
| 执行申请 | 🟡 | 执行程序启动时间 |

**诉讼时效规则（民法典第188条）：** 向人民法院请求保护民事权利的诉讼时效期间为三年。法律另有规定的，依照其规定。诉讼时效中断事件（民法典第195条）包括：向义务人提出履行请求；义务人同意履行义务；权利人提起诉讼或申请仲裁；申请支付令；申请破产；申请诉前保全措施等。

## 模式

此 skill 服务于两种执业设置。从插件配置 CLAUDE.md 中用户的 `## Role` 中选择默认值；用户可以使用标志每次运行时覆盖。

- **`--matter` 模式（内部诉讼律师的默认）。** 以事项历史为中心。从 `matter.md` 阅读事项的案件理论和关键事实，从声明的文档存储来源（飞书云盘、钉钉文档、本地文件系统、iManage——无论 CLAUDE.md 的 `## Landscape` 部分声明什么）提取，并将 `history.md` 视为运行内部日志（决定、保全、保全备忘录——有意不在时间线中）。输出以事项为中心：跨争议发生的事情，标记用于倡导使用。
- **`--documents` 模式（律所律师助理或律师的默认）。** 以证据材料为中心。从配置中读取案件理论，然后从证据清单、当事人文件集或编号的证据材料中提取。输出以证据为中心：材料显示什么，带有证据编号引用，根据案件理论标记。

两种模式汇聚于相同的输出结构（时间线、🔴/🟡/⚪ 重要性标签、差距、SoF 变体）。区别在于来源概况和重要性框架。

如果 `## Role` 是 `solo` 或 `other`，默认为 `--matter` 但在第一次运行时提及两种模式并让用户选择。

## 方框架（重要性标签）

同一事件根据从业者是在证明主张还是在反驳主张而具有不同的重要性。阅读执业档案中的 `## Side`（以及事项覆盖默认值时的逐事项姿态）：

- **原告（进攻性框架）** — 🔴 标记*确立*主张要件（责任、因果关系、损害、通知）的事件，*封闭*被告方试图打开的差距的事件，或以有利于原告的方式*启动*诉讼时效时钟的事件（民法典第188条）。🟡 标记支持主张但可能被质疑的事件。⚪ 是背景上下文。
- **被告（防御性框架）** — 🔴 标记*打破*主张要件（因果关系失败、通知、信赖）的事件，*开启*诉讼时效或管辖权抗辩的事件，或*支持*积极抗辩（免除、弃权、风险承担、比较过错）的事件。🟡 标记削弱原告诉讼叙事的事件。⚪ 是背景。
- **两者兼有 / 因事项而异** — 按时间线询问用户应用哪一方的框架进行重要性标记。基础时间线是中立的；只有重要性解读会改变。

在输出顶部注明应用的框架：`重要性标签从 [原告 / 被告] 视角应用。` 当生成事实陈述变体时，使用方默认值，除非用户另有指定。

## 加载上下文

通用：
- 插件配置 CLAUDE.md → 案件理论上下文（内部法务：`## Landscape` 用于文档来源；律所律师：`## Case theory` 和 `## Document review` 用于平台 + 当事人）、`## Outputs` 用于工作产品标题、`## Decision posture` 用于特权标记规则。
- 此事项的先前 `chronology.md`（如存在）。
- 用户上传的任何文件或会话中提供的路径。

`--matter` 模式还读取：
- `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/[slug]/matter.md` → 案件理论、关键事实、转折事实（用于重要性标记）、关键日期。
- CLAUDE.md 中的默认事项文件夹模式 → 此 slug 的文档存放位置。

`--documents` 模式还读取：
- 证据管理平台元数据（如果有可用的连接器）——按当事人 + 日期范围。
- 证据编号清单或证据目录（如果用户指向一个）。

**冲突门槛——不可绕过（`--matter` 模式）。** 在构建时间线之前，检查 `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/_log.yaml` 中的事项 slug。如果事项不在 `_log.yaml` 中，拒绝并路由：

> "我在事项日志中看不到 [matter slug]。首先运行 `/litigation-legal:matter-intake`，以便冲突检查运行并设置事项工作区。我不会对未接收的事项构建时间线——冲突检查是门槛。"

不要在未接收的事项上继续。`--documents` 模式（在没有事项 slug 的情况下对临时文档集运行）不受此门槛限制，但其输出应被视为诉前研究，不应作为事项工作产品归档。

## 工作流

### 步骤 0：证据保密门槛（每次首先运行）

时间线工作从文档中提取。文档通常受律师职业保密义务约束（律师法第38条）——内部法务事项文件通常默认受职业保密保护；通过调查令或证据交换获得的材料，特别是涉及商业秘密的，可能存在使用限制。从受保密保护的文档中提取内容到后来被共享的时间线中可能造成职业纪律风险，取决于谁接收它以及在什么情况下。

skill 在用户选择保密姿态之前不会提取：

> 在提取之前：来源是如何处理保密性的？
>
> - **A. 所有来源已清除** — 您已经确认这些材料可以用于此目的。我在没有保密标记的情况下提取。
>
> - **B. 混合或尚未确认** — 我提取并用 `conf` 标志标记每个条目：`ok`（来自明确非受保护材料）、`flag`（来自可能受保密保护的材料——律师-客户、工作成果、商业秘密）或 `review`（来源不明确）。标记的条目在输出中视觉标记，事实陈述变体默认过滤掉它们。
>
> - **C. 中止——先确认** — 暂停 skill。确认来源的使用范围。返回并重新运行。

在时间线标题中记录选择为 `confidentiality_posture: A-cleared | B-mixed | C-aborted`。

### 步骤 1：识别文档来源

**`--matter` 模式：**

1. **用户提供的路径** — 本次会话中放入的任何内容（文件路径、飞书云盘链接、钉钉文档导出）。
2. **默认事项文件夹** — 来自 CLAUDE.md 的文档存储模式，为此 slug 展开。
3. **声明的来源** — CLAUDE.md 中的 `Document storage` 表，筛选为此事项可能涉及的（例如，发件方通信的飞书邮件归档、飞书云盘法务文件夹）。
4. **询问** — 如果来源看起来很薄，提示："我可以从我有的内容构建，但时间线将不完整。还有其他要指向我的吗？关键电子邮件/微信记录、合同、内部备忘录、往来函件？"

**`--documents` 模式：**

1. **证据清单 / 编号证据集** — 用户指向证据目录或清单；skill 按证据编号 + 日期读取。
2. **证据管理平台** — 如果 MCP 连接器可用，按当事人 + 日期范围拉取。
3. **当事人文件** — 如果用户提供原始当事人文件或文件夹导出，也读取那些。
4. **询问** — 如果对关键当事人或日期范围的覆盖看起来很薄，提示。

### 步骤 2：拉取 + 阅读

对于每个有可读文件的来源：

- **PDF、电子邮件（.eml）、.docx、.txt** — 直接读取。
- **微信/钉钉聊天记录导出** — 按日期范围 + 关键词查询；否则用户将相关记录导出到文件夹。
- **飞书/钉钉邮件** — 如果 MCP 连接器已认证，按日期范围 + 对方 / 关键词查询；否则用户将相关线程导出到文件夹。

如果 skill 无法访问声明的来源，在输出的差距部分明确命名它，而不是静默继续。

**没有静默补充。** 如果事项某个时期的来源覆盖很薄——声明的时间窗口中文档少于预期——报告发现了什么并停止。不要在未询问的情况下从网络搜索、公共记录搜索或模型知识填补差距。说："来源为 [期间 / 当事人] 返回了 [N] 个事件。覆盖似乎很薄。选项：(1) 指向我的其他来源（证据清单、文件夹），(2) 在网络中搜索此窗口的公共记录事件——结果将标记为 `[web search — verify]`，在依赖前应根据主要来源检查，或 (3) 在此停止并注明差距。你想要哪个？"律师决定是否接受较低可信度的来源；skill 不为他们决定。

**来源归因。** 用事件来源标记每个时间线条目：文件路径、证据编号、MCP 连接器或声明的文档存储来源。对于任何无法追溯到检索文档的事件或日期——例如，从模型训练数据回忆的事实、通过网络搜索找到的公共记录事件——内联标记：`[web search — verify]`、`[模型知识——需验证]` 或 `[用户提供]`（用户在会话中陈述的事实）。标记为 `需验证` 的条目比文档来源条目具有更高的编造风险，应首先检查。

**标记覆盖陈述法律结论、截止期限或计算日期的每个部分——不仅仅是时间线条目。** 时间线来源于文档。差距部分、关键事件部分、理论联系线，以及任何关于诉讼时效（民法典第188条）、时效中断事件、举证期限截止日期、庭审截止或保全裁定的陈述，都是 skill 从模型知识编写的法律分析，除非有来源。每个此类陈述都带有来源标记：`[computed from: <rule cited with tag>]`、`[模型知识——需验证]`、`[用户提供]` 或研究连接器标记。当没有研究连接器可达且 skill 正在计算截止期限或引用规则时，在审阅者注释的 **Sources:** 行中记录。

### 步骤 3：提取事件

对于每个文档，识别有日期的事件：

- **电子邮件/微信/钉钉：** `[日期] [发件方] 告知 [收件方] [主题/内容]`
- **会议：** `[日期] [参会者] 会议讨论 [主题]`（根据会议记录或备忘录）
- **决定：** `[日期] [决策者] 决定 [什么]`（根据记录文档）
- **提交/文书：** `[日期] [当事方] 提交 [起诉状/答辩状/申请]`
- **外部事件：** `[日期] [发生了什么]`（合同签署、产品交付、监管机构行动、事件越过阈值）

通常每个文档一个事件。偶尔零个（无日期或未确立事件）。有时多个（涵盖多个决定的会议摘要）。

**每个条目的保密标记（仅当 confidentiality_posture == B-mixed 时）。三态规则——永远不静默决定主观保密测试未通过：**

- `conf: ok` — 来源**确信**不受保密保护（提交文件、公开信息、无我方律师的对方通信）。仅在不存在合理保密理论时使用。
- `conf: flag` — 来源确信或可能受保密保护（与律师的通信、工作成果备忘录、内部分析草稿、共同代理材料）。**任何不确定事项的默认值** —— 如果主导目的判断很接近，或内容混合，它应该在这里，而不是在 `ok`。
- `conf: review` — 来源表面上不明确，但 skill 无法做出判断（无发件方/收件方元数据、不可读等）。

当 `conf: flag` 或 `conf: review` 时，内联添加 `[SME VERIFY: 保密状态]` 以便律师在审查时看到。标记不足可能放弃保密保护（单向门）；标记过度由律师在审查中纠正（双向门）。优先选择可恢复的错误。

### 步骤 4：去重

同一事件在多个文档中出现：一个会议在三个日历上并产生一封摘要邮件——那是**一个有四个来源的事件**，不是四个事件。合并。合并的条目引用所有来源。

### 步骤 5：标记重要性——按案件理论

从 `matter.md`（`--matter` 模式）或配置的 `## Case theory` 部分（`--documents` 模式）阅读转折事实和关键事实。标记每个事件：

- 🔴 **关键** — 事件是转折事实或对我方有利/不利的关键事实的一部分
- 🟡 **相关** — 上下文、模式证据、支持次要论点
- ⚪ **背景** — 对完整性有用，不会写入文书

**纪律：** 300 个条目都标 🔴 的时间线等于没有标记。将 🔴 保留给真正会影响事实认定者的事件。如果不确定，用 🟡。

**边界标记：** 当条目处于 🔴 和 🟡 之间（或 🟡 和 ⚪ 之间）时，标记为较低重要性并内联添加 `[SME VERIFY — borderline significance call]`。律师的判断将覆盖 skill 的判断。

### 步骤 6：写入

默认输出是工作时间线。按请求提供变体。

### 事实陈述时间线（按请求）

仅筛选 🔴 和相关 🟡。按时间顺序叙述以散文形式呈现——文书事实部分的骨架。每段是一个事件或紧密关联的集群，附有记录引用。

**保密过滤器默认值：** 当 `confidentiality_posture == B-mixed` 时，🔒-标记和 🔒-审查条目默认**排除**。事实陈述变体旨在最终用于外部用途（文书、披露、与对方谈判）——🔒 条目在律师确认保密状态之前不属于那里。如果用户仍然想要包含 🔒 条目，需要明确的 `--include-flagged` 确认；在输出标题中捕获确认作为永久记录。

### 特定证人时间线（按请求）

筛选以命名证人为发件方、收件方、参会者或主题的事件。供证人准备使用，帮助重建证人何时知道什么。

## 与 matter.md / history.md 的集成

**有意分离**（内部法务 `--matter` 模式）。`history.md` 是律师的运行日志——决定、更新、程序里程碑、内部策略笔记。`chronology.md` 是面向倡导的事实时间线。它们重叠但不合并：

- 发出了保全通知 → 进入 history.md（内部行动）。通常不在时间线中（不是争议的事实）。
- 对方在3月14日发送了违约催告函 → 进入 chronology.md（🟡——确立其知情）。如果 intake 引用了它，也在 history.md 中。
- 我们的储备建议备忘录已起草 → 仅在 history.md。

当律师想要将历史事件放入时间线时，可以粘贴它们。默认是保持分离。

## 此 skill 不做什么

- **解决矛盾。** 当两个文档对事件发生时间说法不同时，两个条目都放入并标记。解决是律师的判断；可能需要证人询问或进一步证据收集。
- **发明来源中不存在的事件。** 如果不在文档中（也不在 matter.md 或配置中作为捕获的事实），就不在时间线中——但"差距"可能会将其标记为缺失。
- **保证完整性。** 时间线只与来源一样好。如果证据收集正在进行且只有20%到达，时间线反映这一点。注明限制。
- **为用户决定保密状态。** 步骤 0 门槛强制姿态选择；每个条目的 `conf` 标记捕获首次分类。实际的保密裁定是律师根据 `[SME VERIFY]` 标记的判断。
