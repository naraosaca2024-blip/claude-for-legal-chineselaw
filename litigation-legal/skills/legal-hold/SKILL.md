# /legal-hold

1. 如果 `--status`（无 slug）：阅读 `_log.yaml`，生成组合范围保全报告。
2. 否则：加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/[slug]/matter.md` + 日志行。
3. 加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` → 特权标记、保全模板指针、升级规范。
4. 遵循以下工作流和参考。
5. 按标志路由：
   - `--issue`：捕获范围、保管人、日期范围、系统。起草 `legal-hold-v1.docx`。更新 `legal_hold` 字段。附加历史条目。设置 `next_refresh`（默认 +6 个月）。
   - `--refresh`：捕获范围/保管人变更。起草下一版本。更新 `last_refresh` + `next_refresh`。标记已离职保管人。
   - `--release`：捕获释放日期、保留指令。起草释放通知。设置 `released:` 字段。
6. 在写入前确认。向用户显示起草的通知和日志差异。

# 证据保全通知

## 目的

证据保全通知（内部证据保管通知）是企业内部管理合规的最佳实践文件。在中国，企业面临诉讼时，内部主动通知员工保留相关证据，有助于防止证据灭失，并减少潜在的"证据妨碍"指控风险。此 skill 拥有所有四个阶段：**发布 → 刷新 → （释放）→ 跟踪**。

## 重要的中国法律说明

**中国无法定的企业内部证据保全义务。** 不同于美国的"legal hold"制度（FRCP Rule 37(e)），中国民事诉讼法中**没有**要求企业在诉讼前发布内部保全通知的法定义务。因此：

- 本工具生成的内部保全通知定性为**企业内部最佳实践**，而非法定要求。
- 违反内部保全通知不会自动产生美国法下的"不利推断"（spoliation）制裁。
- **但是：** 中国法律对证据灭失有其自己的处理规则——《最高人民法院关于民事诉讼证据的若干规定》第95条规定，一方当事人控制证据无正当理由拒不提供的，人民法院可推定对方当事人关于该证据内容的主张成立。

**向法院申请证据保全（民诉法第81条）：**
- 这是中国法正式的证据保全机制，向法院申请，由法院依法执行。
- 申请条件：证据可能灭失或以后难以取得，当事人可在诉讼开始后申请；情况紧急时可在提起诉讼前申请。
- 与内部企业通知（本工具）不同——内部通知是自律管理；法院保全是依法强制。

**证据保全与财产保全的区别：**
- **证据保全（民诉法第81条）**：保全证据本身，防止证据灭失。
- **财产保全（民诉法第100-105条）**：冻结财产，防止被执行人转移财产。两者独立申请，目的不同。

## Load context

- `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/_log.yaml` ——日志行（legal_hold 字段 + 状态）
- `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/[slug]/matter.md` ——事项上下文（对手方、事实、来自 internal_owners 的关键保管人）
- `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` ——内部保全模板指针的内部风格、特权标记、升级规范

**冲突门槛——不可绕过。** 在发布、刷新或释放保全通知之前，检查事项 slug 的 `_log.yaml`。如果事项不在 `_log.yaml` 中，拒绝并路由：

> "我在事项日志中看不到 [matter slug]。首先运行 `/litigation-legal:matter-intake`，以便冲突检查运行并设置事项工作区。我不会对未接收的事项发布、刷新或释放证据保全通知——冲突检查是门槛，针对未管理事项发布的通知没有 `_log.yaml` 行来跟踪 `last_refresh` / `next_refresh` / `released`。"

不要在未接收的事项上继续。

## Modes

命令接受一个标志：`--issue | --refresh | --release | --status`。默认（无标志）→ 提示。

### `--issue` — 首次发布

当 `legal_hold.issued == false` 且事项活跃或合理预期时需要。

**在向保管人发布保全通知之前（重大行为）：** 阅读 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` 中的 `## Who's using this`。如果角色是非律师：

> 发布证据保全通知有内部法律合规含义——范围、保管人列表和时间创建企业将在以后争论证据灭失时被评判的内部记录。你是否与律师审查了此内容？如果是，继续。如果否，这是带给他们的简报：
>
> [生成一页摘要：事项和触发、建议的范围和保管人、已知的证据灭失风险、可能出什么问题（太宽/太窄）、要问律师什么。]
>
> 如果你需要在中国找到执业律师：可通过各省市司法局或律师协会官网查询会员，或通过全国律师信息公示平台（www.lawyer.org.cn）查询。

没有明确的是不要发送通知。起草和范围界定不需要门槛——发布需要。

**在发布前研究适用的保全规则。** 识别诉讼阶段和可能适用的保全来源（内部最佳实践、行政监管要求、合同约定保密义务）。如果需要向法院申请正式的证据保全（民诉法第81条），说明两种途径的区别：内部通知 vs. 法院保全申请。

> **内部文件：** 以下通知发送给公司员工。这是内部管理文件，应标注相应的保密级别。

**输入：**
1. **范围**——文件、数据、通信的类别。具体开始：与对手方的合同、所有引用 [项目/主题] 的通信、相关财务记录、日历条目。`[SME VERIFY —范围太宽 = 操作负担；太窄 = 证据灭失风险]`
2. **保管人**——可能持有相关材料的指定个人。从 matter.md internal_owners 和常见角色（业务负责人、人力资源合作伙伴、IT 负责人）获取建议。`[SME VERIFY ——保管人列表是可辩护保全和差距论点之间的区别]`
3. **日期范围**——从何时开始保全（通常是：触发事件或更早），直到现在 + 持续。
4. **系统**——电子邮件、飞书/钉钉/企业微信消息、文件共享、设备（包括适用时的个人设备）、日历、项目管理工具、CRM、旧系统。
5. **紧急性**——如果已送达诉讼或收到带有诉讼威胁的要求，这今天就要发出。
6. **生效日期**——保全的日期。

### `--refresh` — 定期重新确认

刷新节奏：默认 6 个月；可根据事项调整。当 `next_refresh < today`（或用户手动调用）时，skill 起草刷新通知。

**输入：**
1. 自上次刷新以来的任何**范围变更**（新主题、新保管人、新系统）。
2. 任何**要添加或移除的保管人**（离职需要特殊处理——见下文）。
3. 重新确认语言。

**刷新通知模板：** 与发布类似；以"这是最初于 [日期] 发布的证据保全通知的重新确认"开头。列出当前范围（如需要则修改）。请求重新确认。

**已离职保管人：** 如果保管人自上次刷新以来已离开公司，skill 将此标记为保全行动项目——离职员工的文件和电子邮件档案需要在 IT 级别保全，而不仅仅是通过通知个人。在 history.md 中记录为需要行动的单独条目。

**写入：**
- `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/[slug]/legal-hold-v[N].docx`（下一版本号）
- `history.md` 条目
- `_log.yaml`：更新 `last_refresh` 和 `next_refresh` 字段；如变更则修改 `custodians` 列表

### `--release` — 关闭保全

通常在事项关闭时。确认事项真正结束（未上诉、不太可能重新开放、相关主张的诉讼时效已过）。

**在释放保全通知之前（重大行为——保全义务恢复正常状态）：** 阅读 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` 中的 `## Who's using this`。如果角色是非律师：

> 释放证据保全通知有内部法律含义——一旦释放，保管人可能恢复正常文件处置。在错误时间释放会造成证据灭失风险。你是否与律师审查了此内容？如果是，继续。如果否，这是带给他们的简报：
>
> [生成一页摘要：事项状态、为何现在提议释放、相关主张/上诉/诉讼时效敞口、保管人影响、可能出什么问题、要问律师什么。]

没有明确的是不要发送释放通知。

**输入：**
1. 释放权限确认（通常是签署人或总法律顾问）。
2. 释放日期。
3. 保留指令——保全下的材料会发生什么？（恢复正常文件管理？在定义期间内继续保留？转移到档案？）

**释放通知模板：** 一段，正式。"关于 [事项] 于 [日期] 发布的证据保全通知于 [日期] 生效解除。正常文件管理政策恢复适用。"

**写入：**
- `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/[slug]/legal-hold-release.docx`
- `history.md` 条目
- `_log.yaml`：设置 `released: [YYYY-MM-DD]`

### `--status` — 组合范围报告

阅读 `_log.yaml`。生成报告：

```markdown
# 证据保全状态——[今天]

## 活跃保全

| 事项 | 已发布 | 上次刷新 | 下次刷新 | 保管人 | 状态 |
|---|---|---|---|---|---|
| [slug] | [date] | [date] | [date] | [N] | [ok / ⚠️ 刷新到期 / ❌ 逾期] |

## ⚠️ 注意

- **刷新逾期：** [列出 next_refresh < today 的 slug]
- **30 天内刷新到期：** [列表]
- **活跃事项未发布保全通知：** [列表——高/关键风险优先]
- **已关闭事项保全通知仍活跃：** [列表——考虑释放]

## 最近释放

[最近 5 个释放保全通知及其日期]
```

这是一个单独的命令调用（`/legal-hold --status` 无 slug）或由 `/portfolio-status` 调用作为组合汇总的一部分。

## Integration with portfolio-status

`portfolio-status` skill 已经标记"活跃诉讼未发布保全通知"。此 skill 解决这些标记。在打开事项时值得在简报中交叉引用：如果 `legal_hold.issued == false`，`/matter-intake` 通过提议运行 `/legal-hold --issue` 关闭。

## What this skill does not do

- **执行保全。** 它发布通知；IT/保管人保全。skill 在保管人离职时标记（以便 IT 可以在系统级别保全）但不进入系统。
- **替代向法院申请证据保全（民诉法第81条）。** 向法院申请证据保全是正式的司法程序，此 skill 生成的是内部通知，两者不同。如需向法院申请，需要专门的申请书和程序。
- **单独做出范围判断。** skill 从事项上下文提出范围；用户确认。范围太宽 = 操作负担。范围太窄 = 证据灭失风险。用户的判断。
- **未经审查自动刷新。** 即使当 `next_refresh` 到来时，用户在刷新通知发出前审查范围变更。
- **发送通知。** 起草 .docx；用户根据内部惯例通过飞书/钉钉/企业微信或邮件发送。
