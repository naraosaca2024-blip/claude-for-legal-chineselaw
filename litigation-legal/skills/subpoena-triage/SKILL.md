# /subpoena-triage

1. 从提供的路径阅读请求文件。
2. 分类（中国5类分类——见下方）。
3. 如果是监察委/刑事侦查类 → 停止，根据 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` 升级刑事法律顾问。否则继续。
4. 加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/matters/_log.yaml` 进行交叉检查。加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` → landscape、特权惯例、升级规范。
5. 遵循以下工作流和参考。
6. 提取关键字段，分析范围/负担/保密，生成回应框架 + 合规计划 + 截止日历。
7. 编写 `~/.claude/plugins/config/claude-for-legal/litigation-legal/inbound/[slug]/triage.md`。复制或链接文件到 `~/.claude/plugins/config/claude-for-legal/litigation-legal/inbound/[slug]/incoming.[ext]`。
8. 交接：如果未设置保全通知，则 `/legal-hold --issue`；如果重要性需要，则 `/matter-intake`；如果是现有事项中的协查请求，则 `/matter-briefing [slug]`。

# 调查令/协查请求分类

## Purpose

调查令和协查请求带有期限到达。失败模式：错过截止期限、配合过度（超出法定义务的信息）、配合不足（妨碍调查风险）、或错过提出异议的窗口。此 skill 分类、分析并生成带有回应框架的合规计划。

## 中国调查令5类分类

中国法律体系中无法直接对应美国FRCP 45传票的统一制度。以下5类分类是中国实践中最常见的类型，规则和程序实质不同：

### 类型一：法院调查令（法院主动依职权）

**法律依据：** 《民事诉讼法》第67条——人民法院有权向有关单位和个人调查取证，有关单位和个人不得拒绝。

- 由人民法院直接向单位/个人发出
- 被送达方有配合义务，拒绝配合可能构成妨害民事诉讼
- 处理重点：确认法院身份、核实调查范围的合理性、确认是否涉及国家秘密/商业秘密/个人隐私（须申请保密处理）
- PIPL注意：如涉及个人信息向境外提供，须单独分析（详见下方PIPL条款）

### 类型二：律师调查令（当事人委托律师调查取证）

**法律依据及各省差异：** `[模型知识——需验证]`

- **广东省：** 广东省律师协会《广东省律师调查取证操作规范》有较完善规定，调查令由法院签发授权律师向相关单位调查取证。
- **上海市：** 上海市律师协会有相关规范，但法院签发调查令的范围和条件各异。
- **北京市：** 北京高院有相关规定，但司法实践中律师直接持调查令调查的情形较少。
- **其他省份：** 各省实践差异较大，部分省份尚无统一规范，须查询当地法院和律师协会的具体规定 `[模型知识——需验证各省具体规定]`。

处理重点：
- 核实调查令的真实性（法院章、签发法官信息）
- 确认受托律师执业证信息（律师执业证号、所在律所）
- 分析被调查信息是否属于法律保密义务的范围
- 被调查单位对律师调查令**无强制配合义务**，但实务中通常会配合

### 类型三：行政协查（公安/检察院/行政机关）

**法律依据：** 视行政机关性质不同：
- 公安调查协助：《刑事诉讼法》第52-54条
- 金融监管调查：《证券法》第171条、《银行业监督管理法》第34条
- 市场监管调查：《市场监督管理法》等
- 税务协查：《税务稽查工作规程》等

处理重点：
- 区分是否附法律强制力（有强制力：拒绝可能承担法律责任；无强制力：属于行政指导性质）
- 确认发出机关的身份和管辖权
- 分析是否在企业经营范围内的义务
- 涉及刑事侦查的：立即升级法律顾问，高度谨慎

### 类型四：证人出庭申请

**法律依据：** 《民事诉讼法》第73条——知晓案件情况的证人应当出庭作证。

- 一方当事人申请证人出庭，经人民法院通知
- 证人有出庭作证的法定义务（有正当理由除外）
- 拒绝出庭：被申请方可申请法院强制传唤（民诉法第109条）
- 处理重点：确认通知来源（法院通知 vs. 当事人直接请求）、证人的知情范围、是否需要律师协助

### 类型五：监察委-刑事侦查程序

**法律依据：** 《监察法》、《刑事诉讼法》。

- **立即升级刑事法律顾问。** 此类型超出本工具处理范围。
- 监察委留置措施、检察院调查传唤、公安刑事侦查均属此类。
- 任何回应策略均需具备刑事辩护经验的律师主导。

## PIPL第41条：跨境数据特殊规定

**《个人信息保护法》第41条规定：**

> 非经中华人民共和国主管机关批准，个人信息处理者不得向外国司法或者执法机构提供存储于中华人民共和国境内的个人信息。

这意味着：
- **境外法院/监管机构直接要求提供境内个人信息** → 无论以何种形式发出（GDPR协议、美国法院传票、境外监管调查函），均须先经国家主管机关批准，不得直接配合。
- 未经批准直接提供：违反PIPL，可能面临巨额罚款及行政处罚。
- 处理路径：通过外交途径、司法协助条约（如中美司法协助条约）或国家主管机关批准的程序处理。

每次分流中，如涉及向境外提供信息，均须在分流报告中标注此条款。`[模型知识——需核实最新主管机关解释及操作程序]`

## Workflow

### Step 0: Research the applicable rule

**在分析此请求之前，研究该类型调查令/协查的适用法律规则（民事诉讼法、行政法规、监察法等）和具体类型。识别：配合义务的法律性质（强制/指导性）、回应截止期限、保密义务、以及拒绝配合的风险。引用精确参考。验证有效性——规则和当地变体会更改。立即升级刑事侦查类请求给刑事律师。**

**没有静默补充。** 如果对配置的法律研究工具（元典、北大法宝、威科先行、裁判文书网、人民法院案例库）的研究查询为该类型规则、变体或精确引用返回很少或没有结果，报告发现的内容并停止。说："搜索从 [tool] 返回了 [N] 个结果。对于 [rule / type / variant]，覆盖范围似乎很薄。选项：(1) 扩大搜索查询，(2) 尝试不同的研究工具，(3) 搜索网络——结果将标记为 `[web search — verify]`，在依赖前应根据主要来源检查，或 (4) 在此停止。你想要哪个？"律师决定是否接受较低可信度的来源；skill 不为他们决定。

**来源归因。** 在分类输出中，用引用来源标记每条规则引用：`[国家法律法规数据库]`、`[最高人民法院]`、`[元典]`、`[裁判文书网]`、`[人民法院案例库]`、`[北大法宝]`、`[威科先行]`；`[web search — verify]`；`[模型知识——需验证]`；`[用户提供]`。标记为 `需验证` 的引用具有更高的编造风险，应首先检查。

### Step 1: Classify

根据你刚刚研究的规则确认具体类型（参见上方5类分类说明）：

- **类型一：法院调查令** — 由人民法院直接发出，具有法定强制力。
- **类型二：律师调查令** — 由法院授权律师持令调查，各省规则不同。
- **类型三：行政协查** — 公安/检察院/行政机关发出，注意区分有无强制力。
- **类型四：证人出庭申请** — 要求特定员工作为证人出庭作证。
- **类型五：监察委/刑事侦查** — 立即升级刑事法律顾问；不继续标准分类。

### Step 2: Extract key fields

- **发出机构** — 法院（哪个）、行政机关（哪个）、律师（执业证号、律所）
- **被请求方** — 我们公司的哪个实体/个人
- **送达方式** — 邮寄/专人送达/电子送达（对截止日期计算有影响）
- **收到日期** vs. **签署日期**
- **请求类型** — 5类分类中哪一类
- **具体请求** — 他们想要什么、何时
- **回应截止日期** — 根据适用规则
- **地理范围** — 保管人、地点、涉及的系统
- **相关案件** — 请求与哪个诉讼/调查相关

### Step 3: Portfolio cross-check

- **当事方协查 → 与现有事项相关：** 验证标题匹配 `_log.yaml` 中的事项。如果是，路由到该事项的工作流。
- **第三方协查 → 我们不认识的标题：** 捕获当事方；记录为独立传入。
- **来自同一案件的多个请求：** 标记协调发出；可能适用单一回应策略。

### Step 4: Analyze scope, burden, confidentiality

**范围 / 相关性**
- 请求的信息是否映射到我们实际拥有的文件？
- 任何类别是否是捕鱼式调查（过于宽泛、未与基础案件的主张/抗辩相关联）？
- PIPL第41条：是否涉及向境外提供信息？（如有，触发第41条检查）

**负担**
- 涉及的保管人、搜索的系统、时间段
- 估计数量（粗略：小 / 中 / 大 / 极端）
- 成本——第三方协助通常自担费用

**保密性**
- 可能涉及律师职业保密义务（律师法第38条）？（几乎总是是的，对于任何涉及律师的法律工作产品）
- 其他保密——商业秘密、国家秘密、个人信息（PIPL）
- 将需要保密审查——根据 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` 标记格式

**其他异议理由**
- 超出管辖范围
- 程序不合规（送达不规范、主体资格不足）
- 涉及需要保密处理的信息（商业秘密/个人信息/国家秘密）
- 负担过重

### Step 5: Objections framework

起草结构化的回应框架——不是最终回应信，而是适用哪些异议以及原因的大纲。用户（通常与外部律师一起）最终确定。

每个异议：
- 法律依据——引用步骤 0 中研究的规则的精确引用
- 对此请求的具体适用（哪些类别、哪些保管人）
- 强度（强 / 合理 / 弱）

### Step 6: Compliance plan

即使提出异议时，我们通常配合请求的部分内容。计划：

- **可能提供的范围**——异议后，我们要提供什么
- **要搜索的保管人**——姓名和系统
- **日期范围**
- **审查协议**——谁审查保密（我们、外部律师、合规团队）
- **提供格式**——纸质文件/电子文件/系统数据导出

### Step 7: Deadlines

使用步骤 0 研究中确定的截止期限。

- **回应截止日期**——根据研究的规则
- **异议截止日期**——根据研究的规则
- **内部决定截止日期**——我们必须决定的时间（通常是截止日期减5个工作日）
- **PIPL批准时间**（如适用）——申请主管机关批准需要的时间

将所有这些都放入日历。立即行动项目。
