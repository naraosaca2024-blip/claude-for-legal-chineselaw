# /claim-chart

1. 加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` → 角色、工作产品标题、决策姿态、文档存储。
2. 如果事项工作区已启用，确认或选择活跃事项；加载 `matter.md`（立场、管辖法院、阶段、理论、诉状）。
3. 遵循以下工作流和参考。
4. 模式选择：
   - `--patent` → 专利权利要求图表。需要专利号和至少一项主张权利要求。子模式：`--infringement`（侵权）、`--invalidity`（无效）、`--cnipa`（国家知识产权局行政程序）、`--review`（审查）。
   - `--civil` → 民事要件分析表（请求权基础分析）。需要诉因（或抗辩）和立场。
   - 无标志 → 询问用户选择哪个。
5. 民事模式：参阅 skill 目录中的 `references/element-templates.md` 获取基准要件列表。在映射前与用户确认控制性法条或司法解释。
6. 专利模式：将主张权利要求解析为要件，标注有争议的解释术语，应用任何无效宣告决定或法院判决。
7. 将要件映射到目标（被指控产品/现有技术/证据语料库/审查中的图表）。每个单元格精确定位引用。
8. 生成差距列表（民事）或证据需求列表（专利）——这是优先输出。
9. 根据用户偏好写入 markdown、CSV（值 + `_sources` 配套文件）及 Excel 或 Sheets。每个输出附工作产品标题。
10. 如果事项处于活跃状态，写入事项的 `claim-charts/` 文件夹；否则写入执业级 `claim-charts/` 文件夹。如果事项处于活跃状态，向 `history.md` 追加一行。
11. 返回摘要读出：权利要求/诉因、目标、管辖法院、阶段、按状态划分的要件计数、差距列表、文件路径，以及提醒每个单元格都是线索。

# 要件分析表

## 证据材料使用限制

在处理一组诉讼文档之前，询问："这些文档中是否有任何一份是通过法律程序中的调查令、法院查证或证据交换获得的？"如果是：

- **中国民事诉讼：** 依《民事诉讼法》第68条及《最高人民法院关于民事诉讼证据的若干规定》，当事人在诉讼活动中知悉的对方当事人的商业秘密，应当予以保密，不得在本案以外使用。通过法定程序获取的证据材料，应仅用于本案诉讼目的，不得用于其他案件或商业目的。
- **仲裁程序：** 依各仲裁机构规则（CIETAC、上仲等），通过仲裁程序取得的证据材料通常负有保密义务。

确认："此使用在本案诉讼或仲裁程序范围内，且合规。"如果未确认，标注："⚠️ 证据材料可能有使用限制。在继续之前确认此使用是否允许。"

## 图表是草稿，而非发现或主张

**在每个输出顶部放置此内容。不要删除。不要软化。**

> 此图表是供律师分析和核实的草稿，而非已提交的主张、诉状、开庭陈述或法律意见。每个映射都是律师必须对照来源核实的线索。所列要件来自法律规范的构成要件分析——适用法条（民法典、合同法、侵权责任法及各单行法）及相关司法解释始终控制。差距检测是证据收集或动议的起点；它不是关于案件是非曲直的结论。

低报差距是单向门——未具备可信度要素的诉状、未附证据应对有争议要件的文书，或未证明损害赔偿的庭审。高报是双向门——律师在审查中清除标注。默认偏向双向门。

## 模式选择

在任何其他操作之前，在顶部询问：

> 哪种图表？
>
> 1. **专利权利要求图表** — 权利要求限制与被指控产品（`--infringement`）、现有技术（`--invalidity`）或另一方图表（`--review`）的逐要件映射；或向国家知识产权局申请无效宣告（`--cnipa`）的图表。
> 2. **民事要件分析表（请求权基础分析）** — 诉因（或积极抗辩）的法律要件映射到证据。用于诉前分析、举证期限规划、庭审准备。格式：规范依据-构成要件-本案事实-证据-缺口 五列格式。

加上通用 intake（两种模式都需要）：

- **立场。** 主张方还是应诉方？（民事模式中这会翻转举证责任；专利模式中它会翻转侵权/无效框架。）
- **管辖法院/论坛。** 法院（省份+级别）或仲裁机构。各地法院及仲裁规则因机构而异。标注哪个控制。
- **阶段。** 诉前、诉状、举证、庭前会议、庭审准备、庭审后。图表相同；输出的框架不同。
- **现有图表？** 如果是 `--review`，加载它。

# 模式一 — 专利权利要求图表

## 子模式

- `--infringement` — 权利要求要件 vs. 被指控产品（侵权主张、庭审准备）
- `--invalidity` — 权利要求要件 vs. 现有技术（无效抗辩、法院诉讼）
- `--cnipa` — 向国家知识产权局（CNIPA）申请无效宣告（行政程序）
- `--review` — 审查他人制作的图表

## 中国专利无效的两个路径

在中国，专利无效有两个独立路径，规则和程序实质不同：

**路径一：CNIPA行政无效宣告程序**
- 申请机构：国家知识产权局专利局复审和无效审理部
- 申请主体：任何单位或个人（专利法第45条）
- 审查周期：约12-18个月
- 决定效力：国家知识产权局作出的无效宣告请求审查决定，当事人可向北京知识产权法院提起行政诉讼
- 举证标准：准确、充分的证据，合理的无效理由
- 提示：此路径与法院诉讼可并行，但存在协调问题

**路径二：法院诉讼中的无效抗辩**
- 审理机关：北京知识产权法院、上海知识产权法院、广州知识产权法院等
- 抗辩方式：以专利无效为由提起确认不侵权诉讼，或在侵权诉讼中提出中止审理等待CNIPA结果
- 注意：中国法院在侵权诉讼中通常不直接审查专利有效性，会待CNIPA结果后继续

## 专利模式工作流

### 第一步：解析权利要求

将主张的独立权利要求解析为编号要件。处理：

- **前序部分。** 注意它是否具有限制性——这是权利要求解释的问题。标注 `前序-限定性: 待确定`，除非已有相关判决或决定。
- **过渡词语。** "包括"（开放式）/"由……组成"（封闭式）/"主要由……组成"（半开放式）。影响是否有未记载的附加要件会排除侵权。
- **要件** 以逗号/分号分隔，编号为 `[1a]`、`[1b]`、`[1c]`。保持编号稳定——它是图表的脊骨。
- **功能性要件** — 每个"用于[功能]的装置"或非结构性功能术语。范围是说明书中公开的结构加等效物。
- **马库什类型权利要求、从属权利要求** — 用关于不寻常解释规则的说明标注。

向用户显示解析结果。确认后再进行映射。错误的解析会毒害其下的每一行。

### 第二步：权利要求解释检查

标注有争议的术语：

- 说明书中的自创术语或定义术语
- 具有申请历史的术语（修正、论点）
- 功能性语言（"用于……"、"适于……"）
- 相对术语（"大致"、"约"）

对于每个标注的术语，说明在哪种解释下映射有效，在哪种解释下映射失败。如果已有CNIPA决定或法院判决，应用它。

### 第三步：映射

对于每个要件，针对每个目标：

1. **找到证据。** 被指控产品：文件、手册、数据表、源代码、拆解、庭审笔录、专家报告。现有技术：中国专利按说明书页码，公开申请按段落，非专利文献按页/图。
2. **逐字引用。** 逐字逐字。无释义。
3. **描述映射。**

   | 映射 | 含义 |
   |---|---|
   | `literal` | 权利要求语言与被指控特征/现有技术公开相符 |
   | `literal-construction-dependent` | 在X解释下字面符合；在Y解释下不符 |
   | `doe` | 等同（功能-方式-结果） |
   | `anticipation` | 每个要件在单一参考文件中（新颖性缺陷，专利法第22条第2款） |
   | `obviousness-combination` | 第二参考文件提供缺失要件；创造性缺陷（专利法第22条第3款） |
   | `partial` | 要件的部分内容存在 |
   | `not-found` | 要件不存在 |
   | `needs-evidence` | 无法从可用材料判断 |
   | `construction-dependent` | 取决于有争议术语的解释方式 |

4. **每个单元格注明状态。**
5. **标注未解决问题。**

**不得静默补充。** 文件薄意味着 `needs-evidence`，而非从类似产品外推。

### 第四步：从属权利要求——执行，不要敷衍

对于每个主张的从属权利要求，生成实际的行（或一组行）图表附加限制与目标的对应关系。

### 第五步：CNIPA无效宣告特有事项（`--cnipa` 子模式）

标注，不作定论：

- **新颖性缺陷（专利法第22条第2款）** — 现有技术单个文件公开了全部要件
- **创造性缺陷（专利法第22条第3款）** — 主要文件 + 辅助文件 + 组合动机
- **书面描述缺陷（专利法第26条第3款）** — 说明书公开不充分
- **权利要求缺乏支持（专利法第26条第4款）** — 权利要求超出说明书支持的范围
- **不符合专利授权客体（专利法第25条）** — 例如科学发现、智力活动的规则和方法等

无效宣告须以清楚、充分的证据和理由提出。图表中的表面证据在CNIPA审理中不是证明。

# 模式二 — 民事要件分析表（请求权基础分析）

## 中国法请求权基础分析框架

中国民事诉讼中的要件分析以**请求权基础分析**为核心框架。分析步骤：

1. **识别请求权基础** — 原告的诉讼请求建立在哪条法律规范上？（例如：《民法典》第577条违约责任、第1165条过错侵权）
2. **分解构成要件** — 该法律规范的适用需要满足哪些构成要件？
3. **对照本案事实** — 本案具体事实如何对应各构成要件？
4. **列出证据** — 每项构成要件有何证据支撑？
5. **识别缺口** — 哪些构成要件尚无充分证据？

**五列格式（民事模式标准输出格式）：**

| 规范依据 | 构成要件 | 本案事实 | 证据 | 缺口/说明 |
|---|---|---|---|---|

## 工作流

### 第一步：识别请求权基础

- 是什么诉因？（或抗辩？）如果有多个请求权基础，分别建表。
- 哪一方？原告的初步证明案件，被告的积极抗辩，被告对原告初步证明案件的挑战。从执业档案中读取 `## 立场` 获取默认值。
- 哪个管辖法院？省份和级别。**各地法院及仲裁机构的规则不同。**
- 哪份诉状？加载起诉状/反诉状/答辩书，以便图表追踪实际主张的诉因。

### 第二步：加载要件

三条路径：

**(a) 模板库。** 参阅此 skill 目录中的 `references/element-templates.md`。常见诉因和常见积极抗辩的基准要件，附引用法条和司法解释及管辖法院注意事项。

**(b) 自定义。** 用户定义要件，或粘贴法条/司法解释/起诉状中的诉因以供解析。

**(c) 积极抗辩。** 也支持映射抗辩——诉讼时效（民法典第188-196条）、免责条款（民法典第394条）、不安抗辩权（民法典第527条）、情事变更（民法典第533条）等。

**主动提出中国法下的请求权基础建议。** 常见诉因的请求权基础及构成要件：

| 诉因 | 核心请求权基础 | 主要构成要件 |
|---|---|---|
| 合同违约 | 《民法典》第577条 | 合同有效成立；义务内容明确；违约事实；损害后果；因果关系 |
| 侵权损害赔偿 | 《民法典》第1165条（过错侵权） | 行为不法；损害事实；因果关系；主观过错 |
| 不当得利 | 《民法典》第985条 | 他人受损；己方受益；无法律根据；直接因果关系 |
| 产品责任 | 《民法典》第1202条 | 产品缺陷；损害事实；缺陷与损害有因果关系 |
| 劳动合同纠纷 | 《劳动合同法》第80-90条 | 劳动合同存在；违反法定义务；损害；因果关系 |
| 知识产权侵权 | 《著作权法》/《专利法》/《商标法》具体条款 | 权利有效性；侵权行为；损害结果 |

在映射前与用户确认要件列表。如果用户管辖的法院/仲裁机构有特殊要求，询问："您所在法院/仲裁机构的规则是否有增减/改写其中任何一项？"

### 第三步：映射

对于每个要件：

- **支持证据** — 什么能证明这个要件？引用来源并精确定位。
  - 庭审笔录 — `[2026年X月X日庭审笔录第X页]`
  - 声明书/陈述书 — `[证人甲某陈述书第X段]`
  - 书证 — `[证据编号X第X页]`
  - 当事人认可 — `[答辩状第X段认可]`
  - 物证、视听资料
  - 鉴定意见 — `[XX鉴定机构鉴定意见书第X页]`
  - 法条/案例 — 对于纯法律要件
- **逐字引用** 证明性或文件性证据的地方。无释义。
- **反向证据** — 什么往相反方向走？引用。
- **强度** — `strong` / `moderate` / `weak` / `none`。
- **每个单元格注明状态** — `supported` / `partial` / `disputed` / `gap` / `needs-discovery`。

### 第四步：差距检测——关键输出

映射后，生成差距列表：

> **证据薄弱或缺乏的要件：** [列表]
>
> - 如果是主张方（原告）：这些会导致诉讼请求不能得到支持。在下次庭审前填补这些差距。
> - 如果是应诉方（被告）：这些是您的防御焦点和申请驳回的目标。原告必须证明每个要件；差距是一种防御。
> - 如果是举证期限前：这些是您的证据收集重点——将差距转变为 `supported` 或确认 `none` 的调查令申请、证人出庭申请、鉴定申请。

差距检测不是关于案件是非曲直的结论。它是案件薄弱处的地图。

### 第五步：阶段感知框架

询问阶段：

- **诉前/诉状。** 起诉状是否对每个要件做出了可信度陈述？任何仅凭信息和信念主张而无事实支撑的要件都是被告驳回申请的目标。
- **举证阶段。** 对于每个 `gap` 或 `needs-discovery` 要件，需要哪些证据？哪些证人出庭、哪些调查令申请、哪些鉴定申请。
- **庭前会议阶段（民诉法第133条）。** 哪些事实已无争议？对于每个要件，是否存在实质性事实的真实争议？
- **庭审阶段。** 举证顺序。哪位证人证明要件1，哪个证据证明要件2，谁进行质证，基础是什么。

## 此 skill 不做什么

- **不作结论。** 不作侵权、非侵权、责任或不承担责任的结论。永远。
- **不决定权利要求解释**（专利）或**控制性要件**（民事）。它标注有争议术语/基准要件并在陈述的假设下绘制图表。
- **不满足无效的举证标准**或**庭审时的优势证明**。它生成供律师审查的初步草稿。
- **不替代专家分析。**
- **不送达、提交或签署任何内容。** 每个输出都是草稿。
- **不外推。** 如果证据不在，单元格为 `needs-evidence`/`gap`——绝不猜测。
